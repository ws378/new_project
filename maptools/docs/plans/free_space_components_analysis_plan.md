# 自由连通区分析视图设计说明

## 1. 背景

当前 `maptools` 已具备：

- 禁止区
- 虚拟墙
- `pass_only`
- `no_coverage`
- 栅格编辑能力

但在实际使用中，用户很难快速识别：

1. 哪些自由空间是主工作区域
2. 哪些自由空间只是被障碍、未知区、虚拟墙、禁止区切出来的小碎片
3. 哪些小自由区应进一步处理为：
  - 禁止区
  - `no_coverage`
  - 或继续保留

因此需要新增一个“自由连通区分析视图”，把当前工程下的自由空间按连通域拆开、着色，并直接给出面积和像素规模，辅助后续人工治理。

## 2. 目标

本能力的目标不是新增一类长期标注对象，而是提供一套诊断与分析视图，用于：

1. 显示每个自由连通区
2. 统计每个连通区的像素数和面积
3. 帮助识别“小自由区”
4. 为后续人工操作提供依据：
  - 转禁止区
  - 转 `no_coverage`
  - 或保持不变

## 3. 术语

- `free-space component`
  - 自由连通区
- `component mask`
  - 某个自由连通区对应的二值掩膜
- `human-authored polygon`
  - 人工绘制、人工编辑的正式多边形区域
- `algorithm-derived region`
  - 从自由连通区分析结果派生出来的区域语义，默认不是正式多边形真值
- `obstacle connectivity repair radius`
  - 障碍物连通修补尺度，单位米

说明：

- 不直接向用户暴露 `opening/closing` 术语
- UI 层统一使用“障碍物连通修补尺度”

## 4. 输入语义

自由连通区分析前，应先构造“障碍物语义图”。

以下内容都视为障碍：

1. 原始地图中的占据区
2. 原始地图中的未知区
3. 禁止区
4. 虚拟墙

其中：

- 未知区明确按障碍处理
- 禁止区内明确按障碍处理
- 虚拟墙应参与障碍建模，而不是仅做显示层

## 5. 障碍物连通修补

### 5.1 需求含义

用户提出的核心需求是：

- 希望把障碍物孔洞连上
- 或把过窄的小自由缝封住
- 从而更稳定地识别小自由区

从用户语义出发，正式参数定义为：

- `障碍物连通修补尺度 (m)`

默认值：

- `0.5 m`

### 5.2 实现语义

该参数用于在连通域分析前，对障碍物语义做一次“连通修补”。

设计层不强绑底层必须叫 opening/closing，只约束结果语义：

1. 较小障碍孔洞会被连上
2. 较窄自由缝会被封住
3. 自由连通区分析基于修补后的障碍结果进行

### 5.3 与正式规划真值的关系

该参数首先是：

- **分析视图参数**

但在本方案升级后，分析结果还需要支持两类处置动作：

1. 对单个自由连通区右键处置
2. 按面积阈值批量转 `no_coverage`

因此应明确区分：

- **分析过程本身**
  - 不直接修改工程真值
- **用户显式触发的处置动作**
  - 会把结果写回正式工程真值

也就是说：

- 连通区着色和统计本身仍是分析视图
- 但基于分析结果的“转禁止区 / 转 `no_coverage` / 恢复自由区”属于正式编辑动作

## 6. 输出结果

### 6.1 可视化

需要生成自由连通区着色图：

1. 每个连通区独立颜色
2. 连通区边界与底图可同时可见
3. 当前鼠标悬停或选中的连通区需可高亮

### 6.2 统计信息

对每个连通区至少给出：

1. `component_id`
2. 像素数 `pixel_count`
3. 面积 `area_m2`

建议可进一步补充：

4. 外接框尺寸
5. 是否触边
6. 是否包含当前站点/起点
7. 是否与某个 `area_label` 相交

第一版必须项仅为：

- 像素数
- 面积

建议同时输出：

- 是否命中“小自由区 `no_coverage` 阈值”

### 6.3 展示位置

建议在底部状态栏或下方面板显示：

- `component=7, pixels=1823, area=4.56 m²`

当鼠标移到不同连通区时，动态刷新。

## 7. 人工区域与算法区域的真值边界

需要明确区分两类来源：

### 7.1 人工区域

用户通过绘制工具直接创建的区域，本质上是在表达一个 **vector 几何意图**。

因此：

- 人工区域默认就是正式多边形真值
- 可以直接保存为 `constraint_segments`
- 后续编辑也继续以 polygon 方式进行

### 7.2 算法区域

自由连通区分析得到的区域，本质上是 **raster / component / mask 语义**。

用户在这一步真正想表达的是：

- 这块自由区是否禁止
- 这块自由区是否不参与覆盖
- 这块自由区是否恢复为普通自由区

而不是：

- 我要一条精确的正式多边形边界

因此长期上应遵守：

- 算法区域默认不直接固化成正式多边形
- 默认只保存“component 被如何处置”的语义
- 只有在用户显式要求“生成多边形”时，才把该区域轮廓固化成 polygon

### 7.3 当前实现与长期方向

短期为了接入现有 `constraint_segments` 真值链，允许：

- `component mask -> polygon constraint segment`

但应明确这是 **工程接入上的过渡表达**，不是长期最优真值。

长期更合理的方向是：

- 人工区域继续以 polygon 为真值
- 算法区域以 `algorithm-derived region` 为真值
- polygon 仅在用户显式要求时生成

## 8. 产品形态建议

### 8.1 第一版定位

第一版建议做成：

- `View`
  - 或 `Analysis`
    下的一个分析模式

而不是直接塞进常驻 `Edit` 工具组。

原因：

1. 这是诊断能力，不是主绘制能力
2. 第一版不应自动写回工程真值
3. 更适合先作为分析叠加层存在

### 8.2 建议入口

建议提供：

- `Analyze Free Space Components`

可打开：

1. 连通域着色叠加层
2. 参数控制
3. 当前连通区统计

### 8.3 参数

建议提供两个参数：

 - `障碍物连通修补尺度 (m)`
 - `小自由区 no_coverage 阈值 (m²)`

默认：

- `0.5`
- `3.0`

## 9. 连通区处置动作

本功能的主要价值是为后续人工治理提供依据。

需要支持三类动作：

### 9.1 转禁止区

语义：

- 导航和覆盖都不能进入

### 9.2 转 `no_coverage`

语义：

- 导航不一定禁止
- 但覆盖规划时忽略

### 9.3 恢复为自由区

语义：

- 删除当前自由连通区上已存在的 `forbidden_zone` / `no_coverage` 约束
- 让该区域重新回到普通自由区语义

这是和禁止区不同的重要能力，不应混淆。

### 9.4 触发方式

应支持两类触发方式：

1. **单区右键处置**
  - 对当前鼠标所在自由连通区右键
  - 菜单提供：
    - 置为禁止区
    - 置为 `no_coverage`
    - 恢复为自由区
2. **按阈值批量处置**
  - 对面积 `<= 3.0 m²` 的自由连通区
  - 一键批量转 `no_coverage`

### 9.5 多边形生成原则

对算法派生区域，默认处置动作应优先理解为：

- 该 component 被标记为 `forbidden`
- 该 component 被标记为 `no_coverage`
- 该 component 被恢复为 `free`

而不是默认：

- 立刻生成一条正式 polygon 约束

因此产品上建议分成两层动作：

1. **语义处置动作**
  - 置为禁止区
  - 置为 `no_coverage`
  - 恢复为自由区
2. **几何固化动作**
  - 生成多边形
  - 固化成正式可编辑 polygon

长期上，第 1 层应成为默认动作，第 2 层应为显式动作。

## 10. 和现有模型的关系

### 9.1 与禁止区

- 禁止区参与障碍图构造
- 禁止区本身不是分析结果

### 9.2 与虚拟墙

- 虚拟墙参与障碍图构造
- 虚拟墙会影响自由连通域切分

### 9.3 与 `no_coverage`

- `no_coverage` 不应直接当导航障碍
- 在该视图里，它是后续“连通区处置动作”的目标类型
- 因此分析输入阶段不应把 `no_coverage` 直接并入障碍语义图

## 11. 非目标

第一版不做：

1. 自动把小自由区转成禁止区
2. 复杂批量规则系统
3. 和覆盖规划主链强绑定

## 12. 建议的数据输出

如果需要支持调试与后续自动化，建议分析结果内部至少生成：

- `components_overlay`
- `component_stats[]`
- `selected_component_id`

其中 `component_stats[]` 至少包含：

```json
{
  "component_id": 7,
  "pixel_count": 1823,
  "area_m2": 4.56
}
```

## 13. 渐进实施建议

### 阶段 A：只做分析视图

1. 构造障碍语义图
2. 应用障碍物连通修补
3. 提取自由连通域
4. 着色显示
5. 显示像素数和面积

### 阶段 B：接入正式处置动作

1. 对单个连通区右键
2. 支持：
  - 置为禁止区
  - 置为 `no_coverage`
  - 恢复为自由区
3. 通过现有真值链保存处置结果

### 阶段 C：明确人工区域与算法区域分层

1. 人工绘制区域继续直接保存为 polygon
2. 算法区域默认保存为 `algorithm-derived region` 语义
3. 仅在用户显式触发时才生成 polygon

### 阶段 D：批量小自由区处置

1. 新增 `小自由区 no_coverage 阈值`
2. 对面积 `<= 阈值` 的 component 给出候选标记
3. 支持一键批量转 `no_coverage`

### 阶段 E：支持筛选与排序

1. 按面积排序
2. 按面积阈值过滤
3. 快速定位最小若干自由区

## 14. 一句话结论

本方案的核心是：

- 把未知区、禁止区、虚拟墙统一按障碍处理
- 在一个可调的“障碍物连通修补尺度”下重新分析自由空间连通域
- 将每个自由连通区独立着色并显示像素数和面积
- 用分析结果辅助后续人工决定哪些区域应转为禁止区或 `no_coverage`
- 人工区域默认就是 polygon 真值
- 算法区域默认保留为派生区域语义，只有显式要求时才生成多边形
