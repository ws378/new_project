# MapTools 易用性与流程收敛 PRD（One Page）

## 1. 背景
当前工具从“单纯地图编辑”扩展到了“导入-编辑-导出流程工具”，但信息架构与交互仍偏工程化，导致新用户决策成本高、窄屏可用性差、错误恢复困难。

## 2. 本期目标（P0）
1. 在常见分辨率下确保主流程可达（不因布局溢出阻断操作）。
2. 收敛入口，用户无需理解底层格式即可完成完整流程。
3. 缺失可选算法依赖时不阻断主流程，错误提示可执行。

## 3. 用户与核心场景
- 目标用户：地图编辑工程师、算法工程师、交付同事。
- 核心场景：
  - 导入地图或 coverage repo
  - 编辑区域与路径
  - 导出算法可消费产物

## 4. 关键问题
1. 顶部按钮一字平铺，1366x768 下可视区域不足。
2. 打开入口分散（Open Map / Open Project / Import Coverage Repo）导致路径选择困难。
3. 导入导出反馈弱，失败时缺少下一步建议。
4. 可选算法依赖缺失时，历史行为会影响启动稳定性。

## 5. 方案概览
### A. 响应式工具区
- 工具按分组展示：`文件`、`编辑`、`路径`、`标注`、`视图`。
- 断点策略：
  - 宽屏：图标+文字
  - 中屏：图标+Tooltip
  - 窄屏：分组下拉 + “更多”菜单
- 主流程按钮常驻：打开、保存、撤销、重做、导入、导出。

### B. 统一资源打开向导
- 单入口“打开资源”，自动识别：地图 YAML / coverage repo YAML / 项目目录。
- 加载后给出摘要：地图ID、区域数、路径点数、标签数。

### C. 错误与依赖治理
- 启动时做依赖分级提示：必需依赖 / 可选依赖。
- 可选依赖缺失时保留主流程可用，并提示“切换 basic 或配置路径”。
- 错误弹窗统一为：`问题` + `影响` + `操作建议`。

## 6. 范围边界
- 本期不做：
  - 算法能力提升（仅可用性兜底）
  - 大规模视觉重设计
  - 跨进程任务调度

## 7. 验收标准（DoD）
1. 在 1366x768、1920x1080、2560x1440 下，主流程按钮均可达且无截断。
2. 新用户在 10 分钟内可独立完成“导入-编辑-导出”。
3. 缺失可选 planner 依赖时，工具可正常启动且 `basic` 模式可正常生成路径。
4. 导入/导出失败提示中，至少包含一个直接可执行的下一步动作。

## 8. 度量指标
- 首次任务完成率（新用户）>= 90%
- 导入/导出失败后二次成功率 >= 80%
- 布局相关问题反馈下降 >= 70%

## 9. 里程碑
- M1（第1周）：响应式工具区与统一入口原型可用
- M2（第2周）：错误治理与反馈摘要上线
- M3（第3周）：回归测试与文档更新完成

## 10. 决策（本期建议默认值）
1. 默认路径规划模式：`basic`
2. `shelf_aware_guarded` 作为可选增强能力，按需启用
3. 大体积 `output_*` 产物默认不进入主分支（改为样例包或发布产物）
