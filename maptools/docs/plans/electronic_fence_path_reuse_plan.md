# 电子围栏约束分段对 Path 交互的复用方案

## 1. 目的

本文件补充说明：

1. 当前 `path` 相关交互能力中，哪些适合复用到电子围栏/约束分段
2. 哪些能力不能直接复用
3. 哪些代码可以抽成公共层
4. 电子围栏约束分段还需要新增哪些能力

本文件只讨论：

- 交互复用
- 模型边界
- 演进方向

不在这里直接确定具体类名和最终代码拆分细节。

## 2. 总结结论

电子围栏约束分段应当：

- 像 `path` 一样编辑
- 像 `annotation` 一样承载业务语义

也就是：

1. 点列编辑交互可大量参考 `path`
2. 数据模型不能直接复用 `CoveragePathNode` / `CoveragePathManager`
3. 约束类型、闭合状态、导出语义必须保持独立

## 3. 当前 path 体系里可直接借鉴的能力

### 3.1 点列编辑能力

电子围栏约束分段和 `path` 一样，天然都是“有序点列”。

因此下面这些能力可以直接借鉴：

1. 点级拖拽
2. 点级增删
3. 段级选中
4. 边级插点
5. 多段共存显示
6. 选中态高亮
7. 点序保持

### 3.2 命中测试能力

电子围栏分段的编辑需要：

1. 命中某个点
2. 命中某段边
3. 命中整条段

这和 `path` 的交互目标接近，可以直接参考其命中逻辑。

### 3.3 分段与连接思路

`path` 当前已经是多段组织，不是单条无结构 polyline。

这意味着可以借鉴：

1. 多段对象管理
2. 单段局部修改不影响其它段
3. 某条段的增量重绘

### 3.4 可视化风格组织

`path` 已经有：

1. 基础线
2. 点
3. 选中态
4. 标签

电子围栏也可以沿用这种分层渲染思路，只是颜色和语义不同。

## 4. 当前 path 体系里不应直接复用的部分

### 4.1 覆盖路径业务字段

电子围栏约束分段不应直接复用这些字段：

1. `room`
2. `segment`
3. `yaw`
4. `acc_dist`
5. `room_dist`
6. `seg_dist`
7. 任何“覆盖路径累计距离”语义

这些属于覆盖规划结果，不属于约束对象。

### 4.2 路径管理器语义

当前 `CoveragePathManager` 的目标是：

- 保存覆盖路径结果
- 维护路径点序和路径文件

而电子围栏约束分段的目标是：

- 保存约束几何
- 保存闭合状态
- 保存约束类型

因此不应把电子围栏做成 `CoveragePathManager` 的一个模式。

### 4.3 导入导出语义

`path` 目前仍带有：

- TSV
- coverage repo 路径保存

这些都不适合继承到电子围栏约束段。

电子围栏应走工程真值和正式导出映射，不应重走路径文件流。

## 5. 适合抽成公共层的能力

### 5.1 Polyline / Polygon 编辑基元

长期建议抽出一层公共编辑基元，用于服务：

1. path
2. 约束分段
3. area label
4. forbidden/pass_only 等未来统一模型

这一层可以包含：

1. 点级命中
2. 边级命中
3. 拖点
4. 插点
5. 删点
6. 分段切开
7. 闭合状态切换
8. 简单重绘辅助

### 5.2 几何辅助函数

建议抽共用能力：

1. 最近点到线段距离
2. 命中阈值判断
3. polygon/polyline 取点
4. 线段切开
5. 开口段闭合
6. 闭合段打断

### 5.3 画布交互状态机

如果后续要同时维护：

- path
- constraint segment
- area polygon

则建议抽出统一的编辑状态机概念，例如：

1. selecting
2. dragging_point
3. dragging_segment
4. inserting_point
5. splitting_segment
6. creating_open_segment
7. creating_closed_segment

## 6. 电子围栏约束分段还需要新增的能力

这是和 `path` 最不同的部分。

### 6.1 闭合状态

约束分段需要正式字段：

- `closed: bool`

而 path 当前没有这套正式语义。

### 6.2 约束类型

约束分段需要正式字段：

- `constraint_type`

至少支持：

1. `forbidden_zone`
2. `virtual_wall`
3. `no_coverage`
4. `pass_only`

### 6.3 类型约束校验

系统应强制：

1. `virtual_wall` 必须 open
2. `forbidden_zone` 必须 closed
3. `no_coverage` 必须 closed
4. `pass_only` 必须 closed

这套校验是 `path` 没有的。

### 6.4 切段后的语义延续

当一条段被切开后，需要明确：

1. 两段是否继承原类型
2. 闭合段被切开后是否自动变 open
3. 名称是否继承
4. 是否需要重新编号

### 6.5 线类与区类互转

需要支持：

1. open -> closed
2. closed -> open
3. 切换类型时的合法性校验

这套行为在 `path` 中也不存在。

## 7. 建议的演进顺序

### 第一批：交互复用

目标：

1. 不先重写全部画布交互
2. 先复用或借鉴 path 的点列编辑方式
3. 做出电子围栏分段的最小可编辑能力

建议包括：

1. 新建 open segment
2. 新建 closed segment
3. 选中段
4. 拖点
5. 插点
6. 删点

### 第二批：约束语义

目标：

1. 加入 `constraint_type`
2. 加入 `closed`
3. 加入类型合法性校验

### 第三批：高级编辑

目标：

1. 切开
2. 合并
3. 线区互转
4. 单段类型切换

### 第四批：导出与工程真值统一

目标：

1. 工程保存恢复统一写 `constraint_segments`
2. 导出映射改成从 `constraint_segments` 出发

## 8. 对现有代码的具体判断

### 8.1 可以重点参考的对象

当前最值得参考的是：

1. `maptools/tools/path_tools.py`
2. `maptools/views/map_canvas.py` 中对路径和点的显示方式
3. 与路径相关的点/段命中思路

### 8.2 不建议直接复用的对象

不建议直接复用：

1. `CoveragePathNode`
2. `CoveragePathManager`
3. TSV 保存链
4. coverage repo 路径保存链

### 8.3 当前“载入电子围栏”的判断

当前 `load_electronic_fence(...)` 加载的是：

- `trajectory_from_tf.jsonl`
- `summary.json`

并写入：

- `reference_trajectory_world`

所以它更接近：

- 参考轨迹层

而不是：

- 正式约束分段对象

这条链不适合作为新电子围栏模型的直接基础。

## 9. 一句话结论

电子围栏约束分段应当大量复用 `path` 的点列编辑交互，但必须保持独立的数据模型、闭合语义和约束类型语义。

可以复用的是：

- 编辑方式
- 命中方式
- 多段交互经验

不能复用的是：

- 覆盖路径业务字段
- 覆盖路径保存链
- 覆盖路径结果语义
