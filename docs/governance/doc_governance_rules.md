# 文档治理规则

## 长期文档允许内容

根层长期文档只允许保存：

- 总索引。
- 稳定架构、算法边界、数据契约和产品入口设计。
- 当前代码入口、脚本生命周期和资产规则。
- 已确认但未解决的缺口。
- 验证矩阵、基线口径和验收记录。
- 已稳定决策、原因和影响范围。
- 历史参考归档。

## 默认删除或吸收内容

以下内容不得作为根层长期文档保留：

- 执行流水账。
- 弱门禁聊天记录。
- 过期阶段计划。
- 临时讨论和恢复卡。
- 已被正式设计吸收的过程记录。
- 只描述“本轮”“本阶段”“当前批次”的文档。
- 没有维护入口的生成资产清单。
- 无证据结论或重复审查。

删除前必须确认其中的稳定事实已经吸收到正式文档。

## 新增文档准入

新增文档前必须先判断归属：

- 仓库级长期治理事实：放入 `docs/governance/`。
- 仓库级验证矩阵和验收口径：放入 `docs/validation/`。
- 可重复执行的跨机器操作：放入 `docs/operations/`。
- 历史材料：放入 `docs/archive/`，并明确“不可作为当前运行入口”。
- 子工程自己的需求、设计、quickstart、专题计划：放入对应子工程文档目录。

## 命名规则

- 文件名应直接表达长期职责，不用“阶段1/阶段2/第N批/开工前/执行记录”命名长期文档。
- 标题必须与文件职责一致。
- README 只能做入口索引，不能用索引掩盖下层职责混乱。

## 提交说明要求

涉及地图编辑、路径编辑、转弯代价、覆盖路径、地图导入导出或交付地图处理的文档或代码变化时，提交或 PR/MR 描述应补充：

- `影响范围`
- `验证情况`
- `证据路径`
- `关联跟踪`

如果没有对应算法交付跟踪任务，应明确写“未关联”。
