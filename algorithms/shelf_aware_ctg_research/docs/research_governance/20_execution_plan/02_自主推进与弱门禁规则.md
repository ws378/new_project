# 自主推进与弱门禁规则

## 1. 自主推进回答

可以自主推进完成本研究目录的相关工作，但推进边界如下：

- 可以自主完成文档、诊断脚本、研究版实验脚本、可视化、summary、测试和提交。
- 可以按阶段自主提交，提交范围必须清楚。
- 不在未授权时修改正式 `shelf_aware_guarded` planner。
- 不恢复独立 axis path。
- 不把 output 大图和批量 run 默认提交。
- 遇到重大算法方向冲突、破坏性操作、用户工作区冲突或无法自行补齐的外部输入时，必须停下确认。

## 2. 每批执行顺序

```text
确认批次范围
检查工作区
确认输入 run / case
实施脚本或文档
运行最小验收
检查输出结构
回写执行记录
按边界提交
继续下一批
```

## 3. 弱门禁口径

当前环境不强制启动子 agent。若后续明确要求多 agent，弱门禁角色为：

- 结果审查：检查图、JSON、summary 是否互相一致。
- 代码审查：检查是否混入新策略、独立 axis path 或正式 planner 修改。
- 测试审查：检查 run 是否覆盖重点区域和回归区域。

弱门禁不阻塞主线，但意见必须进入执行记录。

## 4. 过程记录要求

每批至少记录：

```text
批次
阶段
目标
范围 / 非范围
输入
实施结果
验收结果
结论
下一步
提交
```

## 5. 停止条件

只有以下情况主动停下：

- 用户要求改正式 planner，但当前证据不足以定义安全边界。
- 需要删除或重写用户已有数据。
- 当前 run 或 case 缺失导致结论不可复现。
- 诊断结果推翻已确认方向，需要重新定方案。
