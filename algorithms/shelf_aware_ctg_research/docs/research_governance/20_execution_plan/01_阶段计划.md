# 阶段计划

## 1. 执行原则

按“先诊断，再模块优化”的顺序推进。每一阶段必须有：

- 输入 run 或固定 case。
- 输出图和 JSON。
- 可复跑脚本。
- 文档记录。
- 明确非范围。

## 2. A：新研究目录基线记录

目标：确认 `shelf_aware_ctg_research` 是干净、可信的研究起点。

要做：

- 对 `fourfloor`、`beiguo_lanshan_0407` 固定跑 baseline。
- 记录 run 目录、area 数量、核心图完整性。
- 确认没有 CTG-guided 组合算法，没有独立 axis path。
- 写入 project summary 和执行记录。

当前状态：基本完成，后续可补更结构化的 baseline report。

目标：让路径、territory、junction、jump 在图上可解释。

输出建议：

```text
02_edge_switch_points.png

```

重点区域：

```text

```

## 4. C：grid node 质量诊断

目标：解释节点质量问题是否导致远跳。

需要分析：

- grid node degree。
- 是否贴边。
- cell 内 free pixel 面积。
- 是否来自 `complete_cell_test()` 的中心调整。
- 是否孤立或半孤立。
- 是否处于小凸起或边界毛刺。
- fallback 最后访问节点是否属于低价值节点。

输出建议：

```text
nodes_debug_enriched.png
grid_node_metrics.json
low_value_node_overlay.png
residual_node_overlay.png
```

## 5. D：残留点与远跳原因分析

目标：回答“为什么为了一个或几个点跳很远”。

输出建议：

```text
jump_diagnostics.json
jump_overlay.png
jump_target_context.png
```

必须包含：

- jump 距离。
- jump 起终点。
- jump target 对应 node。
- territory label。
- degree。
- nearby free area。
- 到已有路径距离。
- 是否可能被实际清扫宽度覆盖。

## 6. E：形成残留点处理判据

目标：先形成规则，不急着改算法。

分类：

```text
required
optional_by_overlap
optional_by_small_area
optional_by_boundary
snap_candidate
insert_candidate
```

判据必须写成可复核条件，不能只写主观判断。

## 7. F：研究版 grid node pruning / residual handling

目标：在研究目录实现第一个模块优化，不进入正式 planner。

可选方向：

- 规划前标记低价值 optional nodes。
- fallback 前判断远跳目标是否可跳过。
- 规划后删除或合并为少量残留点产生的远跳段。

验收：

- 覆盖损失在容忍范围内。
- 路径长度下降。
- 远跳次数下降。
- 可视化上不再为了边界毛刺绕行。

## 8. G：territory / axis 接入评估

目标：基于 B/C/D/E/F 的证据，决定 territory / axis 是否接入。

可能接入点：

- grid node 分类。
- fallback 约束。
- candidate energy soft prior。
- edge group / corridor group 诊断。

当前不做强约束。
