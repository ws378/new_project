# 连接合法性规则重述

本文用于重述 `channel_topology_graph` 中 node 内连接、directed lane 候选、以及后续 sweep 连接的统一合法性原则。

本文目标不是立刻改代码，而是先把下面三件事说清楚：

- 什么连接允许。
- 什么连接不允许。
- 为什么旧规则在 `dead_end` 与纯回环场景下不够。

后续关于 `virtual_junction`、纯回环切口、`dead_end` 折返 candidate、coverage 接管策略，均以本文作为连接合法性的基线约束。

## 1. 目标

当前系统需要统一解决三类场景：

- 普通路口节点：多条边在节点内汇集，存在 through / turn / 选择关系。
- `dead_end` 节点：只有一条真实边，但允许折返。
- `virtual_junction` 节点：由纯回环切口等人工引入的虚拟节点，需要承接 through 语义。

这三类场景后续都要落到同一条主线上：

- 节点内连接候选。
- directed lane candidate。
- directed lane graph。
- coverage / sweep 连接。

因此必须先统一“连接合法性”的判断口径。

## 1.1 适用范围

本文覆盖的对象包括：

- `junction_rebuild` 阶段输出的节点与边。
- `topology_graph_build` 阶段的 node-local connection candidate。
- directed lane candidate / directed lane graph。
- coverage 阶段对 topology 连接的继承与细化。

本文暂不直接规定：

- `virtual_junction` 的最终字段结构。
- 纯回环切几刀、切口如何落点。
- `dead_end_return` 与 `controlled_u_turn` 的最终执行策略。

这些属于下一轮设计，但它们都不能违反本文定义的连接合法性约束。

## 2. 顶层原则

后续所有连接判断统一遵守：

- 只允许真经过边的连接。
- 不允许假经过边的连接。

这里的关键不是 `in_edge` 和 `out_edge` 是否相同，而是：

- 这个连接是否对应一条可解释的、真实的边级 traversal。
- 它是否真的走过了应经过的边段。
- 它是否只是借节点关系在局部做了一个拓扑假穿越。

## 3. 基本定义

### 3.1 真经过边

指该连接可以被解释为：

- 从某条真实边进入节点局部区。
- 在节点局部区完成合法连接。
- 再从某条真实边离开。
- 或在特殊节点中，沿同一真实边完成真实折返。

核心要求是：

- 被连接到的边必须被真正 traversal。
- 不能只在节点区“改一下方向”就声称经过了边。

### 3.2 假经过边

指连接在拓扑关系上看似成立，但实际上：

- 没有真正走完应经过的边。
- 跳过了中间应被 traversal 的边段。
- 只是利用节点局部邻接关系，把不该直接连接的两段拼起来。

### 3.3 真连接与假连接的判定关注点

后续判断一个连接是否合法，不能只看“图上能不能连”，而要同时看下面三个问题：

- 这个连接是否对应真实边的进入与离开。
- 这个连接是否遗漏了本应 traversal 的边。
- 这个连接是否只是节点局部的拓扑拼接，而没有真实通道语义支撑。

## 4. 为什么旧规则不够

当前存在一个粗规则：

- `in_edge == out_edge` 直接非法。

这条规则的历史作用是防止一部分“假经过边”。但它的问题是：

- 它把“同边相关的真折返”也一并杀掉了。
- 无法支持 `dead_end`。
- 无法支持纯回环切口后的合法 through / return 结构。
- 它判断的是“边 id 是否相同”，不是“是否真经过边”。

因此后续不能再把这条规则当成最终连接合法性定义，只能把它看作过去的一条粗近似保护。

## 5. 普通节点的合法性规则

普通节点指真实场景中的正常汇交节点，不含人工切口语义。

普通节点内允许的连接必须满足：

- 连接从一条真实入边进入。
- 连接切换到另一条真实出边离开。
- 中间节点局部路径可解释。
- 不跳过应 traversal 的边。

普通节点内默认不允许：

- 仅依赖节点局部关系，把同一条边原地翻面。
- 借节点关系跨过应经过的边。
- 形成没有真实边 traversal 支撑的局部闭环。

因此对普通节点来说：

- `same-edge self transition` 默认非法。
- through / left / right 等连接必须基于真实的入边和出边。

### 5.1 普通节点中的反例

下面这类连接应视为非法：

- 从 `edge_1` 进入节点，只在节点内部跨过 `edge_2` 的汇交点，却直接接到 `edge_3`。
- 从 `edge_1` 的一侧进入，又在同一节点内切回 `edge_1` 的另一方向，但并未形成该边上的真实折返语义。

这类情况的共同问题是：

- 表面上“经过了节点”。
- 实际上没有形成真实边 traversal。
- 本质上是假经过边。

## 6. dead_end 节点的合法性规则

`dead_end` 节点只有一条真实 incident edge。它的特殊性在于：

- 它不具备普通节点那种“从一条边进、换另一条边出”的结构。
- 但它允许真实折返。

因此 `dead_end` 节点必须支持一种特殊的真经过边连接：

- 沿唯一真实边进入端点。
- 在 dead_end 局部完成折返。
- 再沿同一真实边反向离开。

其语义不是：

- 节点里原地把 `edge` 当成自己转给自己。

而是：

- 这条边被真实 traversed 了一次进入、一次离开。
- 是真实 return，不是假连接。

所以 `dead_end` 节点允许“同一真实边相关”的连接，但必须满足：

- 它被标记为 `return / u_turn / dead_end_turnback` 这一类特殊语义。
- 它对应真实 traversal，而不是节点内自循环。

### 6.1 dead_end 的设计约束

`dead_end` 节点虽然只有一条 incident edge，但它不能被系统当作“没有 keep candidate 的异常点”直接跳过。原因是：

- 从 coverage 角度看，`dead_end` 是真实要处理的端点。
- 从通道遍历角度看，它需要把“进”和“回”组织成一个合法的边级连接。

因此 `dead_end` 不应再被理解为：

- “没有连接”。
- “只能在 coverage 阶段临时补丁式处理”。

更合理的定义是：

- 它在 topology 层就应拥有一种合法的、被显式标注语义的折返连接。

## 7. virtual_junction 节点的合法性规则

`virtual_junction` 是人工引入的虚拟节点，用于表达例如纯回环切口。

它的作用不是伪造场景语义，而是：

- 把原本系统无法表达的结构，重新组织成主线可消费的拓扑。

对 `virtual_junction`，应优先采用：

- 选取切口。
- 把原回环切成两段真实 edge。
- 这两段 edge 都 incident 到该虚拟节点。

这样该节点处的合法连接就可以自然表达为：

- 一段真实 edge 进入。
- 另一段真实 edge 离开。

这属于 through 语义，是合法的真经过边连接。

对 `virtual_junction` 不应默认理解成：

- `dead_end`。
- 原地折返。
- 只有特殊掉头语义的节点。

它首先是一个“承接 through 的结构性节点”。

### 7.1 virtual_junction 的最小语义要求

不管后续纯回环切口最终采用哪种表达方式，`virtual_junction` 至少要满足：

- 它是显式节点，而不是隐式特例。
- 它的引入不能破坏现有普通节点语义。
- 它承接的连接必须仍然满足“真经过边”原则。
- 它的连接类型需要能被 sweep 层识别，而不是只在 topology 层短暂存在。

## 8. 同边相关连接的统一判定

后续不再使用下面这种粗口径作为最终判断：

- `in_edge == out_edge` 就非法。
- `in_edge != out_edge` 就合法。

而改为：

- 若连接涉及同一真实 edge，两种情况分开判断。

第一种：普通节点中的同边局部翻面。
结论：默认非法。

原因：

- 通常没有形成真实 traversal。
- 高概率是假经过边。

第二种：特殊节点中的同边真实折返。
结论：可以合法。

前提：

- 节点类型允许，例如 `dead_end`。
- 连接语义被明确标记为 `return / u_turn`。
- 能解释成真实 traversed path，而不是局部假连接。

### 8.1 一个更准确的替代口径

后续不应再把“是否同边”当作最终合法性判断，而应改成下面这套口径：

- 是否发生了真实入边。
- 是否发生了真实出边。
- 若入边与出边关联到同一条真实 edge，这个过程是否代表一条真实折返路径。
- 该折返是否属于当前节点类型允许的连接语义。

这样才能同时覆盖：

- 普通节点上的非法同边假翻面。
- `dead_end` 上的合法真折返。
- 纯回环切口后的特殊但真实的边级连接。

## 9. directed lane candidate 的语义要求

一个 directed lane candidate 不是单纯的拓扑配对，而必须对应：

- 一个可执行的 node-local 连接语义。
- 一个明确的 `connection_kind`。

后续建议把 candidate / lane 的语义显式区分为至少这几类：

- `through`
- `turn`
- `dead_end_return`
- `cycle_through`
- `u_turn` 或 `controlled_u_turn`

这样做的目的不是增加复杂度，而是避免把所有合法连接都塞进一个模糊的 `keep` 里。

也就是说：

- `keep` 只表示“允许进入正式图”。
- 但它内部必须再区分连接语义类型。

### 9.1 为什么不能只保留 keep / non-keep

如果只保留 `keep` / `non-keep` 二值标签，会有两个问题：

- `dead_end_return` 与普通 through 都会被压平成“keep”，下游无法区分。
- coverage 层无法知道某个连接到底是直行、正常转向，还是需要风险约束的折返类动作。

因此后续即便维持现有主线对象不大改，也至少应补充：

- `connection_kind`
- 必要时补充 `node_type`
- 必要时补充“是否虚拟节点 / 虚拟连接”的说明字段

## 10. sweep 连接的继承规则

sweep 连接必须继承同一原则：

- 只允许真经过边的 sweep 连接。
- 不允许假经过边的 sweep 连接。

这意味着：

- 拓扑层如果允许了某类特殊连接，coverage 层不能把它当成普通 through。
- 例如 `dead_end_return` 必须在 coverage 层继续被识别为“折返类连接”。
- 后续风险控制、碰撞代理、是否允许 controlled u-turn，都基于这个语义继续判断。

因此 topology 与 sweep 之间不能只传“连得上”这个事实，还要传：

- 连的是哪类连接。
- 是否虚拟。
- 是否属于折返类 traversal。

### 10.1 coverage 层不应再做的事情

coverage 层不应自行把一个 topology 层本来非法的连接补成合法，尤其不应：

- 看到某节点走不通，就在 sweep 层随意给一个回头动作。
- 忽略 topology 连接语义，把所有 keep 都等价处理。
- 把 `dead_end_return` 当作普通 through 使用。

coverage 层只能在 topology 已经放行的合法连接集合上做进一步筛选，例如：

- 是否允许 `controlled_u_turn`
- 是否存在碰撞风险
- 是否满足局部空间约束

## 11. 对现有系统的最小兼容要求

为了不破坏当前已有结果，后续规则改造必须满足：

- 普通节点的 through / turn 结果保持现有口径不变。
- 只有在 `dead_end` 与 `virtual_junction` 这类特殊节点上，才引入新的连接合法性分支。
- 旧 case 中已有的正常 `keep candidate` 不应因为新规则被大面积重分类。
- 新增字段应优先用于补充语义，而不是重写旧对象主干结构。

因此推荐做法是：

- 保留现有主线对象。
- 在 candidate / lane 上新增语义字段。
- 在特殊节点上增加局部规则，而不是全局放开 `same-edge`。

## 11.1 对实现改造顺序的约束

后续改造建议遵守下面顺序：

1. 先重述并固定连接合法性规则。
2. 再设计 `virtual_junction` / 纯回环切口的拓扑表达。
3. 再补 `dead_end` 的真折返 candidate。
4. 最后再让 coverage 接管这些新语义。

这样做的原因是：

- 如果先改 coverage，不改 topology，容易形成补丁链。
- 如果先全局放开 `same-edge`，会把旧的“假经过边”保护直接打穿。
- 先把语义说清，再做对象字段与执行路径改造，风险最小。

## 12. 最终规则摘要

后续连接合法性可归纳为以下 5 条：

1. 合法连接的核心标准是“真经过边”，不是“edge_id 是否相同”。
2. 普通节点只允许真实入边到真实出边的 through / turn 连接，不允许同边局部假翻面。
3. `dead_end` 节点允许真实折返，但该折返必须被显式建模成特殊连接语义，不能伪装成普通 through。
4. `virtual_junction` 节点用于把原本不可表达的结构组织成可消费拓扑，其主要职责是承接真实 through。
5. topology 层与 sweep 层必须共享同一连接合法性语义，不允许上游放行“假经过边”、下游再硬拼接。

## 13. 当前讨论结论

截至本文版本，已经明确的结论有：

- 不能再用 `in_edge == out_edge` 作为最终合法性定义。
- “真经过边”是统一判断口径。
- `dead_end` 需要进入主线，而不是继续作为被跳过的异常分支。
- 纯回环后续需要显式拓扑表达，但具体采用哪种切口组织方式，下一轮再定。

尚未在本文内定稿的内容有：

- `virtual_junction` 的最终数据结构。
- 纯回环切一刀还是两刀，以及对 edge 数量的影响。
- `dead_end_return` 与 `controlled_u_turn` 的精细边界。
- coverage 层对折返类连接的最终接管策略。
