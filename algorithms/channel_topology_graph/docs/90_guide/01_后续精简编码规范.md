# 后续精简编码规范

## 1. 目的

本规范用于约束 `channel_topology_graph` 后续所有“代码瘦身 / 结构清理 / 旧链路收口”类改动。

目标不是单纯减少文件数或代码行数，而是：

- 删除假层级、假抽象、镜像层、重复逻辑
- 保留稳定职责边界和长期可维护结构
- 降低调试成本、阅读成本、误改成本
- 保证主线语义、注释力度、真实 case 行为不退化

本规范优先级高于“为了快先并一处再说”的临时处理习惯。

## 2. 总原则

后续精简必须同时满足下面四条：

1. 主线更短
2. 真值更少
3. 结构更稳
4. 注释不弱

禁止把“删文件”误当成“结构变好”。

如果只是把多个文件机械并进一个大文件，导致：

- 单文件越来越大
- 职责边界更模糊
- 调试入口更难找
- 注释丢失或失配

这种不算合格精简。

## 3. 精简优先顺序

后续所有精简按下面顺序推进，不允许跳着做。

### 3.1 第一优先级：删假层级

优先删除下面这些结构：

- 只做一次字段转发的包装层
- 只做一次对象镜像的中间层
- 只提供单个调用点的薄 helper 文件
- 只为兼容历史命名而存在的镜像对象
- 只为旧 writer / debug 留存，但主线已不再消费的投影层

判定标准：

- 是否产生新的正式语义
- 是否形成稳定职责边界
- 是否被多个独立主消费点真正复用

如果三条都不满足，应优先删除或并回主消费点。

### 3.2 第二优先级：收重复逻辑

优先收掉下面这些重复实现：

- 相同的 route / segment 组装逻辑
- 相同的 through 合法性判断
- 相同的 transition 查找 / 索引查找
- 相同的 reachability 递归
- 相同的评分 key / 排序 key 组装

要求：

- 收成一套正式实现
- 不允许“主线一套、repair 一套、debug 一套”
- 不允许“旧函数保留，新增函数再来一套”

### 3.3 第三优先级：再压局部 helper

只有在前两步完成后，才允许压下面这些内容：

- 只在单文件内使用一次的 helper
- 只为少量分支存在的包装器
- 只做参数重排的局部函数

这里的原则是：

- 单次使用、语义不独立的 helper，优先内联
- 多次使用、语义稳定的 helper，保留

## 4. 文件职责约束

### 4.1 单文件行数要求

后续新增或重构后的单文件，默认约束如下：

- 常规文件目标：`<= 250` 行
- 可接受上限：`<= 300` 行
- 超过 `350` 行，必须主动审视是否拆分
- 超过 `400` 行，原则上视为结构异常，必须给出拆分理由

禁止把“我删掉了几个文件”作为单文件超长的理由。

### 4.2 拆分依据

文件拆分必须按“稳定职责”拆，不允许按“函数类型”机械拆。

正确的拆分方式示例：

- `cadence_actions`: 动作枚举、动作合法性、动作落状态
- `sweep_cadence_build`: greedy 主求解
- `route_repair_pipeline`: route 后处理总控
- `through_semantics`: through 规则与 route 试探

错误的拆分方式示例：

- `helpers_a.py`
- `helpers_b.py`
- `utils_transition.py`
- `misc.py`

### 4.3 允许的“总控文件”

可以保留少量总控文件，但必须满足：

- 只负责主流程编排
- 不堆积大量局部几何细节
- 不堆积多个独立职责的大段实现

如果一个总控文件同时承载：

- 主流程编排
- 几何判定
- 路径组装
- repair 评分
- 多种对象转换

则已经不再是总控文件，而是职责坍塌。

## 5. 注释规范

### 5.1 注释强度不能下降

精简过程中，注释力度必须保持，不允许用以下方式制造“表面净删”：

- 删除关键注释
- 删掉解释性段注释
- 文件搬移后不搬注释
- 把原来明确的注释缩成一句空话

禁止出现下面这种行为：

- 代码搬了，注释没搬
- 旧注释删了，新位置没补
- 为了减少行数，把说明性注释直接清空

### 5.2 注释应该绑在真值点上

注释应优先绑定到下面这些位置：

- 正式主入口
- 正式真值对象组装点
- 关键合法性判断
- 关键排序 / 打分入口
- 会影响行为边界的 fallback 分支

不要把大量核心注释绑在准备删除的薄包装层上。

### 5.3 注释风格要求

注释要回答下面四类问题之一：

- 这段逻辑解决什么真实问题
- 为什么必须这样做，不能直接简化
- 这里的边界条件是什么
- 这里和相邻阶段的职责边界是什么

不需要保留纯废话注释，例如：

- “给变量赋值”
- “调用函数”
- “遍历列表”

## 6. 主线真值约束

### 6.1 一层正式真值

每个阶段只允许保留一层正式真值对象。

禁止长期并存：

- 主对象一套
- 镜像对象一套
- 为兼容保留的旧对象一套
- writer/debug 再复制一套

如果确实还存在兼容链路，必须满足：

- 兼容层不参与新逻辑主消费
- 兼容层不再继续扩字段
- 兼容层有明确删除计划

### 6.2 不允许“镜像投影继续长大”

如果某对象只是对正式真值做重排、改名、复制，就不允许继续往里面新增：

- 新字段
- 新规则
- 新调试语义
- 新下游依赖

否则会重新形成第二真值。

## 7. 精简时禁止的做法

后续严禁下面这些做法：

### 7.1 先并成超大文件，再长期不拆回去

临时收口可以接受，但必须很快拆回稳定职责边界。

不能形成下面这种状态：

- 文件更少了
- 但关键文件膨胀到 500~800 行
- 阅读入口比以前更差

### 7.2 只做重命名，不删结构

比如：

- 旧对象还在
- 旧字段还在
- 旧调用链还在
- 只是名字更好看

这种不算精简。

### 7.3 为了兼容而继续加壳

禁止新增：

- `legacy_xxx`
- `compat_xxx`
- `adapter_xxx`

除非该层是短期必要且有明确删除时点。

如果当前需求就是“旧对象没用了”，那就直接删，不要再包一层“兼容入口”。

### 7.4 一次批量脚本改很多文件，不做即时校验

允许用脚本批量改，但每一轮必须立刻做：

- 读回关键文件
- `compileall`
- 单测

否则很容易出现“半改状态”“旧 import 残留”“注释错位”。

### 7.5 用删空行、删注释制造净删指标

净删只能来自：

- 删除薄层
- 删除重复逻辑
- 合并正式真值
- 压缩无意义 helper

不能来自：

- 删注释
- 删必要空行
- 降低可读性

## 8. 建议的实施流程

后续每轮精简建议严格按下面顺序进行。

### 8.1 第一步：画主链

先明确这一轮要处理的主链，例如：

- `candidate -> cadence -> final_path`
- `route merge -> singleton absorb -> endpoint repair`

只处理一条主链，不要同时对多个 unrelated 区域下刀。

### 8.2 第二步：识别假层级

列出这一条主链上：

- 哪些文件只是转发
- 哪些对象只是镜像
- 哪些 helper 只是参数重排

先删这些。

### 8.3 第三步：合并重复逻辑

把同一语义的多套实现收成一套。

要求：

- 只保留一个正式入口
- 旧入口必须删除，不允许并存

### 8.4 第四步：恢复结构边界

如果中间为了收口临时并大了文件，这一步必须重新拆回：

- 稳定职责文件
- 300 行以内目标
- 注释就位

### 8.5 第五步：验证

每轮至少做下面三层验证：

1. `compileall`
2. 相关单测
3. 真实 case

如果是主线行为变动，还必须补：

- 至少一组批量 case
- 关键可视化结果自查

## 9. 验证口径

### 9.1 基础验证

- `python3 -m compileall -q ...`
- `python3 -m pytest -q ...`

### 9.2 主线验证

如果改动 `candidate -> cadence -> final_path` 主链，必须跑真实 case。

至少包括：

- 单 case 验证
- 5 case 批量验证

### 9.3 可视化验证

如果改动影响：

- 几何路径
- sweep 连接
- route repair
- final path connector

则必须自查可视化输出，不允许只看测试通过。

## 10. 后续推荐结构

针对 `coverage_planning/sweep_cadence`，后续推荐长期收敛到下面这种结构：

- `cadence_actions.py`
  负责动作枚举、动作合法性、动作落状态
- `sweep_cadence_build.py`
  负责 greedy 主求解与 cadence build 正式入口
- `route_repair_pipeline.py`
  负责 route merge / singleton absorb / endpoint repair
- `cadence_kernel.py`
  只保留上下文类型、稳定共享工具、预算配置读取

要求：

- 单文件尽量控制在 300 行上下
- 不再新增薄包装文件
- 不再新增镜像对象层

## 11. 审查清单

每轮提交前，至少自查下面这些问题：

1. 是否真的删掉了薄层，而不是只换名字
2. 是否删掉了重复逻辑，而不是换了位置继续保留两套
3. 是否把注释完整迁移到新的真值点
4. 是否有文件被并得过大
5. 是否还残留旧 import、旧 helper、旧对象引用
6. 是否做了 compile、单测、真实 case
7. 是否有为了临时过关加入的新兼容壳、新 fallback 壳

如果有任一项答案不明确，这轮精简就不能算完成。
