# 整体方案总览

## 1. 文档目的

本文档把当前源码中的正式主线对象流放在一张图上，明确回答三件事：

1. 端到端四个 stage 分别输入什么、输出什么。
2. `CoveragePlanning` 内部当前真实落地的 build-info 对象流是什么。
3. 哪些字段已经是正式 contract，不能继续按旧口径描述。

本文档只描述当前源码已经正式落地的对象流，不再保留旧版平铺字段、旧版 stage_a/stage_b 命名、也不把 `route_sweep_order` / `ordered_passes` 继续写成“未来态”。

## 2. 端到端对象流

当前正式对象流是：

```text
raw input
    -> GeometryPreparationResult
    -> JunctionRebuildResult
    -> TopologyGraphBuildResult
    -> CoveragePlanningResult
```

其中四个 stage 级正式输出分别是：

1. `GeometryPreparationResult` - 第 1 步基础几何准备的正式结果对象。
2. `JunctionRebuildResult` - 第 2 步交汇重建的正式结果对象。
3. `TopologyGraphBuildResult` - 第 3 步拓扑图构建的正式结果对象。
4. `CoveragePlanningResult` - 第 4 步覆盖规划的正式结果对象。

## 3. 各步输入输出

## 3.1 第 1 步 `geometry_preparation`

输入：

- 原始地图灰度图与地图 yaml 元信息。
- 区域约束。
- 几何裁剪与骨架相关配置。

输出：

- `GeometryPreparationResult`
  - `gray` - 运行尺度下的原始灰度图，是后续几何与骨架处理的基础底图。
  - `region_mask` - 参与本次规划的区域约束掩膜，限定算法有效工作范围。
  - `free_mask` - 运行尺度下的自由空间掩膜，是后续通行与覆盖判断的基础真值。
  - `obstacle_mask` - 运行尺度下的障碍物掩膜，用于和自由空间形成互补几何世界。
  - `after_open_mask` - 形态学开操作后的自由区结果，用于稳定后续骨架与拓扑提取。
  - `skeleton_mask` - 原始骨架掩膜，记录自由区中心线的初始提取结果。
  - `skeleton_pruned_mask` - 裁剪和修枝后的骨架掩膜，是后续重建更稳定的骨架输入。
  - `debug_info` - 面向排查和可视化核对的调试信息集合。
  - `validation_info` - 面向契约闭环检查的校验结果集合。

## 3.2 第 2 步 `junction_rebuild`

输入：

- `GeometryPreparationResult` - 第 1 步基础几何准备的正式结果对象。

输出：

- `JunctionRebuildResult`
  - `node_info_list` - 第 2 步输出的正式节点列表。
  - `edge_info_list` - 第 2 步输出的正式边列表。
  - `debug_info` - 面向排查和可视化核对的调试信息集合。
  - `validation_info` - 面向契约闭环检查的校验结果集合。

## 3.3 第 3 步 `topology_graph_build`

输入：

- `JunctionRebuildResult` - 第 2 步交汇重建的正式结果对象。

输出：

- `TopologyGraphBuildResult`
  - `graph_info` - 第 3 步首次建立的正式图对象，是主线图真值承载体。
  - `incident_port_info` - 第 3 步生成的正式 incident port 派生层信息。
  - `node_local_connection_hypothesis_info` - 第 3 步构建的正式 node-local connection hypothesis 派生层，供第 4 步转移候选展开使用。
  - `debug_info` - 面向排查和可视化核对的调试信息集合。
  - `validation_info` - 面向契约闭环检查的校验结果集合。

## 3.4 第 4 步 `coverage_planning`

真实入口：

- `algorithms/channel_topology_graph/stages/coverage_planning.py`
- 入口函数：`build_coverage_plan(...)`

输入：

- `TopologyGraphBuildResult` - 第 3 步拓扑图构建的正式结果对象。
- `geometry_preparation_result`
  - 通过 `context` 继续传入。
  - 对应第 1 步基础几何准备的正式结果对象。
- coverage planning 私有配置。

输出：

- `CoveragePlanningResult`
  - `graph_info` - 第 4 步继续引用和投影后的正式图对象。
  - `coverage_lane_sweep_info` - 第 4 步前半段的正式 build-info，收口 coverage lane 与 sweeps 真值。
  - `sweep_graph_build_info` - 第 4 步 sweep 图层的正式 build-info，收口 group、port view、candidate 和 sweep graph。
  - `sweep_cadence_build_info` - 第 4 步 cadence 子域的正式 build-info，收口 cadence 路线和覆盖统计。
  - `final_coverage_path_build_info` - 第 4 步 final path 子域的正式 build-info，收口最终路径真值与校验结果。
  - `debug_info` - 面向排查和可视化核对的调试信息集合。
  - `validation_info` - 面向契约闭环检查的校验结果集合。
  - `meta` - 阶段级元信息与关键计数摘要。

## 4. `CoveragePlanning` 内部真实对象流

当前源码里 `build_coverage_plan(...)` 的正式对象流是：

```text
topology_graph_build_result.graph_info
+ topology_graph_build_result.node_local_connection_hypothesis_info
+ geometry_preparation_result
    -> coverage_lane_sweep_info
    -> graph_info(attach_edge_coverage_info 后的 graph 视图)
    -> sweep_graph_build_info
    -> sweep_cadence_build_info
    -> final_coverage_path_build_info
    -> CoveragePlanningResult
```

其中四个 coverage 子域 build-info 的职责分别是：

1. `coverage_lane_sweep_info`
   - 契约职责：收口 `coverage_lane_info + sweeps + summary + validation_info`。
   - 正式回答：哪些 edge 被解释成 coverage lane、每条 coverage lane 上生成了哪些正式 sweeps、lane+sweep 子域的最小校验结果是什么。
2. `sweep_graph_build_info`
   - 契约职责：收口 `sweep_group_info + sweep_port_view_info + sweep_transition_candidate_info + sweep_graph_info + summary`。
   - 正式回答：sweep 如何分组、节点口如何排序、哪些候选会收口成正式 sweep graph。
3. `sweep_cadence_build_info`
   - 契约职责：收口 `sweep_cadence_info + coverage_stats + summary`。
   - 正式回答：正式 cadence routes 是什么、`route_sweep_order / ordered_passes` 是什么、覆盖率是什么。
4. `final_coverage_path_build_info`
   - 契约职责：收口 `final_coverage_path_info + summary + validation_info`。
   - 正式回答：如何把 cadence 物化成最终扫线段与路口连接段路径点链，以及最终路径是否合法。

这里不再使用旧版平铺口径：

- 不再把 `CoveragePlanningResult` 写成 `coverage_lane_info / sweeps / sweep_group_info / ...` 顶层平铺。
- 不再把 `CoveragePlanningResult` 写成旧版四阶段命名平铺结构。
- 当前单一真值是四个 build-info 子对象。

## 5. 各层对象契约职责

## 5.1 `coverage_lane_sweep_info`

契约职责：

- 作为第 4 步前半段的正式 build-info，统一承载 `coverage_lane_info`、`sweeps`、`summary`、`validation_info`。

正式承载字段：

- `coverage_lane_info` - 边级 coverage 解释结果集合。
- `sweeps` - 当前 coverage lane 下生成的正式 sweep 对象集合。
- `summary` - lane 与 sweep 子域的规模摘要。
- `validation_info` - lane 与 sweep 子域的闭环校验结果。

闭环关注点：

- `coverage_lane_info[*].source_edge_id` 能回指正式 `graph_info.edges[*]`。
- `coverage_lane_info[*].sweep_ids` 能回指正式 `sweeps[*].sweep_id`。
- `sweeps[*].coverage_lane_id` 能回指正式 `coverage_lane_info[*].coverage_lane_id`。

## 5.2 附着后的 `graph_info`

契约职责：

- 继续承载第 3 步建立的正式图对象。
- 在不重建图对象语义的前提下，把 coverage lane 边级真值投影到 `edges[*].coverage_info`。

正式承载字段：

- `nodes` - 正式图中的节点对象集合。
- `edges` - 正式图中的边对象集合。
- `meta` - 图级元信息与 coverage 投影留痕。
- `edges[*].coverage_info` - 边级 coverage 投影真值。

闭环关注点：

- `graph_info` 仍然只承载图真值，不直接承接 sweeps、cadence 或 final path。
- `edges[*].coverage_info.sweep_ids` 能回指 `coverage_lane_sweep_info.sweeps`。

## 5.3 `sweep_graph_build_info`

契约职责：

- 作为 sweep 图层的正式 build-info，统一承载 group、port view、candidate、sweep graph 和 summary。

正式承载字段：

- `sweep_group_info` - sweep 分组真值。
- `sweep_port_view_info` - 节点口排序视图真值。
- `sweep_transition_candidate_info` - sweep 转移候选真值。
- `sweep_graph_info` - cadence 求解直接消费的正式 sweep 图对象。
- `summary` - sweep 图层规模摘要。

闭环关注点：

- `sweep_group_info.groups[*].ordered_sweep_ids` 能回指正式 `sweeps`。
- `sweep_port_view_info.items[*].ordered_port_sweep_ids` 能回指正式 `sweeps`。
- `sweep_graph_info.connection_units[*].out_sweep_global_id / in_sweep_global_id` 能回指正式 `sweeps`。

连接真值来源约束：

- 正式主口径优先看 `connection_units`。
- 只有旧数据或兼容读取场景下，才回退到 `transitions`。

## 5.4 `sweep_cadence_build_info`

契约职责：

- 作为 cadence 子域的正式 build-info，统一承载 `sweep_cadence_info`、`coverage_stats`、`summary`。

正式承载字段：

- `sweep_cadence_info` - 最终 sweep 顺序骨架。
- `coverage_stats` - cadence 覆盖统计结果。
- `summary` - cadence 子域规模摘要。

闭环关注点：

- `route_sweep_order` 是否与各 route 的 `sweep_sequence` 一致。
- `ordered_passes` 是否与 route `segments` 一一对应展开。
- `coverage_stats` 是否完整覆盖正式 sweep 全集。

正式 contract 说明：

- `route_sweep_order` 已经是正式 contract 字段，不是临时视图。
- `ordered_passes` 已经是正式 contract 字段，既回写到 route 内部，也在 cadence 全局平铺保留。

## 5.5 `final_coverage_path_build_info`

契约职责：

- 作为 final path 子域的正式 build-info，统一承载 `final_coverage_path_info`、`summary`、`validation_info`。

正式承载字段：

- `final_coverage_path_info` - 最终路径物化真值。
- `summary` - 最终路径规模摘要。
- `validation_info` - 最终路径契约级校验结果。

闭环关注点：

- `ordered_items` 与 `junction_connections / routes` 是否一致。
- `junction_connections[*].connection_id` 是否全局唯一。
- `final_coverage_path_info.routes` 是否与 cadence routes 对齐。
- final path 校验信息是否与 summary 规模一致。

## 6. 第 4 步内部伪代码

```python
def build_coverage_plan(topology_graph_build_result, config, context):
    geometry_result = require_geometry_preparation_result(context)

    coverage_lane_sweep_info = build_coverage_lane_sweep_info(
        graph_info=topology_graph_build_result.graph_info,
        geometry_result=geometry_result,
        config=config,
    )

    graph_info = attach_edge_coverage_info(
        graph_info=topology_graph_build_result.graph_info,
        coverage_lane_info=coverage_lane_sweep_info.coverage_lane_info,
    )

    sweep_graph_build_info = build_sweep_graph_info(
        graph_info=graph_info,
        node_local_connection_hypothesis_info=topology_graph_build_result.node_local_connection_hypothesis_info,
        coverage_lane_sweep_info=coverage_lane_sweep_info,
    )

    sweep_cadence_build_info = build_sweep_cadence_info(
        coverage_lane_sweep_info=coverage_lane_sweep_info,
        sweep_graph_build_info=sweep_graph_build_info,
        config=build_sweep_cadence_config(config),
    )

    final_coverage_path_build_info = build_final_coverage_path_info(
        graph_info=graph_info,
        geometry_result=geometry_result,
        coverage_lane_sweep_info=coverage_lane_sweep_info,
        sweep_graph_build_info=sweep_graph_build_info,
        sweep_cadence_build_info=sweep_cadence_build_info,
        config=build_final_coverage_path_config(config, geometry_result),
    )

    return CoveragePlanningResult(
        graph_info=graph_info,
        coverage_lane_sweep_info=coverage_lane_sweep_info,
        sweep_graph_build_info=sweep_graph_build_info,
        sweep_cadence_build_info=sweep_cadence_build_info,
        final_coverage_path_build_info=final_coverage_path_build_info,
        ...,
    )
```

## 7. 当前对象依赖关系

当前应按下面方式理解对象依赖：

1. `node_info / edge_info`
   - 契约位置：第 2 步建立身份，第 3 步收束进 `graph_info`。
   - 依赖关系：第 4 步继续只读消费，不再重建节点边身份。
2. `coverage_lane_sweep_info`
   - 契约位置：建立在 `graph_info + geometry_result` 上。
   - 依赖关系：为 sweep graph、cadence、final path 提供上游 sweep 真值。
3. `sweep_graph_build_info`
   - 契约位置：建立在 `coverage_lane_sweep_info + graph_info + node_local_connection_hypothesis_info` 上。
   - 依赖关系：为 cadence 提供正式 sweep graph 真值。
4. `sweep_cadence_build_info`
   - 契约位置：建立在 `coverage_lane_sweep_info + sweep_graph_build_info` 上。
   - 依赖关系：为 final path 提供唯一节拍来源。
5. `final_coverage_path_build_info`
   - 契约位置：建立在 `graph_info + geometry_result + coverage_lane_sweep_info + sweep_graph_build_info + sweep_cadence_build_info` 上。
   - 依赖关系：作为第 4 步最终路径真值收口对象。

## 8. 三类闭环校验

## 8.1 静态闭环

至少检查：

- `edge_info` 与 `node_info` 回指是否一致。
- `coverage_lane_sweep_info.sweeps` 是否能回指 `coverage_lane_info`。
- `sweep_graph_build_info.sweep_graph_info` 中的连接真值是否能回指正式 sweep。

## 8.2 节拍闭环

至少检查：

- `route_sweep_order` 是否与各 route 的 `sweep_sequence` 一致。
- `ordered_passes` 是否与 route `segments` 一致。
- `coverage_stats` 是否完整覆盖正式 sweep 全集。

## 8.3 结果闭环

至少检查：

- `final_coverage_path_info.routes` 是否与 cadence routes 对齐。
- `junction_connections` 是否全局编号唯一。
- route seam 是否连续。
- final path 校验信息是否与 summary 规模一致。

## 9. 总结

当前整体方案的关键不是把对象写少，而是把正式对象流写准：

- 第 3 步正式输出 `graph_info + incident_port_info + node_local_connection_hypothesis_info`。
- 第 4 步正式输出 `CoveragePlanningResult`。
- `CoveragePlanningResult` 当前正式结构是：
  - `coverage_lane_sweep_info` - 第 4 步前半段 build-info，总结 coverage lane 与 sweeps。
  - `sweep_graph_build_info` - 第 4 步 sweep 图层 build-info。
  - `sweep_cadence_build_info` - 第 4 步 cadence 子域 build-info。
  - `final_coverage_path_build_info` - 第 4 步 final path 子域 build-info。
- `route_sweep_order` / `ordered_passes` 已经是正式 cadence contract。
- FinalCoveragePath 的连接真值主口径优先来自 `connection_units`，不是只看旧版 `transitions`。

这才是当前源码真正落地的整体方案总览。
