# 总体设计说明

## 1. 文档目的

本文档只做一件事：

- 用当前源码的真实对象链，把 `plan1` 主线设计讲清楚

这里强调两点：

1. 本文档以现有源码为准，不另起一套对象世界
2. 语义收口只收术语和层级，不改算法逻辑

如果要看：

- 具体字段契约
  - 去看 `4_核心对象契约整理.md`
- 某一步具体怎么做
  - 去看对应方案文档

## 2. 当前四步主线

当前源码主线仍然是四步：

1. 基础几何准备
2. 交汇重建
3. 拓扑图构建
4. 覆盖规划

四步正式结果对象分别是：

1. `GeometryPreparationResult` - 第 1 步基础几何准备的正式结果对象。
2. `JunctionRebuildResult` - 第 2 步交汇重建的正式结果对象。
3. `TopologyGraphBuildResult` - 第 3 步拓扑图构建的正式结果对象。
4. `CoveragePlanningResult` - 第 4 步覆盖规划的正式结果对象。

也就是说，当前主线首先要收口的是：

- 阶段结果对象

而不是凭文档先发明新的并列一级对象。

## 3. 四步分别回答什么问题

## 3.1 第 1 步：基础几何准备

回答：

- 运行尺度下的几何世界是什么

正式输出：

- `geometry_preparation_result` - 第 1 步基础几何准备的阶段结果变量。

它提供：

- `gray` - 运行尺度下的灰度底图。
- `free_mask` - 运行尺度下的自由空间掩膜。
- `obstacle_mask` - 运行尺度下的障碍物掩膜。
- 骨架及其修剪结果 - 自由空间中心线及其修枝后的结果真值。
- `resolution_m_per_px` - 像素到米制的分辨率基准。

## 3.2 第 2 步：交汇重建

回答：

- 主拓扑层的节点和边是什么

正式输出：

- `node_info_list` - 第 2 步输出的正式节点列表。
- `edge_info_list` - 第 2 步输出的正式边列表。

这一层已经把：

- 节点身份 - 每个节点的唯一身份与基础属性已经被确定。
- 边身份 - 每条边的唯一身份与端点关系已经被确定。
- 节点 polygon - 节点区域多边形几何已经稳定下来。
- 边路径 - 边的正式路径几何已经稳定下来。

稳定下来。

## 3.3 第 3 步：拓扑图构建

回答：

- 第 2 步的节点和边如何聚合成正式图对象
- `incident_port_info` 与 `node_local_connection_hypothesis_info` 如何作为正式 topology 派生层组织

正式输出：

- `graph_info` - 第 3 步首次建立的正式图对象。
- `incident_port_info` - 第 3 步生成的正式 incident port 派生层。
- `node_local_connection_hypothesis_info` - 第 3 步生成的正式 node-local connection hypothesis 派生层。

这里最重要的边界是：

- `graph_info` 当前只承接 `node_info + edge_info` - 也就是正式图只收节点和边，不提前混入第 4 步 sweep 真值。
- sweep 级对象并不在这一步写进 `graph_info` - sweep 相关真值仍属于第 4 步覆盖规划范围。

## 3.4 第 4 步：覆盖规划

回答：

1. 每条边在覆盖意义上如何解释
2. 应生成哪些 sweeps
3. 这些 sweeps 之间有哪些可用转移
4. 最终 sweep 顺序怎么走
5. 最终路径点链是什么

正式输出都打包在：

- `CoveragePlanningResult` - 第 4 步覆盖规划阶段的总结果对象。

其内部当前源码主线是：

1. `coverage_lane_info` - 边级 coverage 解释层真值。
2. `sweeps` - 正式生成的 sweep 列表真值。
3. `sweep_group_info` - sweep 分组真值。
4. `sweep_port_view_info` - 节点口排序视图真值。
5. `sweep_transition_candidate_info` - sweep 转移候选真值。
6. `sweep_graph_info` - cadence 直接消费的 sweep 图真值。
7. `sweep_cadence_info` - 最终 sweep 节拍骨架真值。
8. `final_coverage_path_info` - 最终路径物化真值。

## 4. 第 4 步的语义分层

第 4 步对象多，但语义上可以收成四层：

## 4.1 边级覆盖解释层

- `coverage_lane_info` - 每条边在覆盖意义上的解释结果集合。

它回答：

- 每条 `edge` 在覆盖意义上如何被解释和使用

## 4.2 sweep 静态与粗筛图层

- `sweeps` - 所有 coverage lane 上生成出的 sweep 集合。
- `sweep_group_info` - sweeps 按 coverage lane 与端点关系聚合后的组信息。
- `sweep_port_view_info` - 各组 sweeps 在节点口的空间排序视图。
- `sweep_transition_candidate_info` - sweep 到 sweep 的转移候选粗筛结果。
- `sweep_graph_info` - cadence 求解直接消费的 sweep 图对象。

它们共同回答：

- 有哪些 sweep
- 同一 coverage lane 内如何分组
- 在节点口怎么排序
- 哪些 sweep-to-sweep 转移值得保留

## 4.3 节拍层

- `sweep_cadence_info` - 基于 sweep graph 求出的最终 sweep 顺序骨架。

它回答：

- 最终 sweep 顺序骨架是什么

当前源码里它不是 `ordered_items` 结构，而是：

- `routes` - cadence 结果中的 route 列表。
  - 每条 route 含：
    - `sweep_sequence` - 当前 route 中 sweep 的访问顺序。
    - `transition_sequence` - 当前 route 中 transition 的经过顺序。
    - `segments` - 当前 route 中 sweep 之间连接原语的压缩段序列。
    - `start_end_type` - 当前 route 起始端的端点类型。
    - `end_end_type` - 当前 route 结束端的端点类型。

## 4.4 最终路径层

- `final_coverage_path_info` - 最终 sweep 段与路口连接段已经物化后的路径结果。

它回答：

- 在 cadence 给定骨架上，最终 sweep 段与路口连接段的路径点链是什么

## 5. 当前真正的依赖方向

当前源码依赖方向是：

```text
geometry_preparation_result
    -> node_info_list + edge_info_list
    -> graph_info + incident_port_info + node_local_connection_hypothesis_info
    -> coverage_lane_info + sweeps
    -> sweep_group_info + sweep_port_view_info + sweep_transition_candidate_info + sweep_graph_info
    -> sweep_cadence_info
    -> final_coverage_path_info
```

这个顺序不能在文档里被改写成另一套更“理想”的流程。

## 6. 当前主线伪代码

```python
def run_plan1_pipeline(raw_input, config):
    geometry_preparation_result = build_geometry_preparation(raw_input, config)

    junction_rebuild_result = build_junction_rebuild(
        geometry_preparation_result,
        config,
    )

    topology_graph_build_result = build_topology_graph(
        junction_rebuild_result,
        config,
    )

    coverage_planning_result = build_coverage_plan(
        topology_graph_build_result,
        config,
        context={
            "geometry_preparation_result": geometry_preparation_result,
        },
    )

    return {
        "geometry_preparation_result": geometry_preparation_result,
        "junction_rebuild_result": junction_rebuild_result,
        "topology_graph_build_result": topology_graph_build_result,
        "coverage_planning_result": coverage_planning_result,
    }
```

这段伪代码只做一件事：

- 把当前源码的阶段依赖顺序原样写清楚

## 7. 这轮文档收口的边界

当前允许收口的是：

1. 把第 4 步对象按语义层分清
2. 把同一层里的术语解释清楚
3. 把源码里真实的主流程、分支和依赖写清楚

当前不允许做的是：

1. 把 `coverage_lane_info` 文档里偷改成别的对象
2. 把 `sweep_graph_info` 偷拆成源码里不存在的独立 `connection_unit_info`
3. 把 `sweep_cadence_info` 的 `segments / sweep_sequence` 偷写成另一套 route contract
4. 把 `final_coverage_path_info` 的真实展开逻辑改写成另一套理想流程

## 8. 总结

当前总设计应理解成：

1. 四步主线不变。
2. 第 4 步 stage 级正式结构已经收成：
   - `coverage_lane_sweep_info` - 第 4 步前半段 build-info，总结 coverage lane 与 sweeps。
   - `sweep_graph_build_info` - 第 4 步 sweep 图层 build-info。
   - `sweep_cadence_build_info` - 第 4 步 cadence 子域 build-info。
   - `final_coverage_path_build_info` - 第 4 步 final path 子域 build-info。
3. 第 4 步子域内部仍然会继续使用：
   - `coverage_lane_info` - 每条边在覆盖意义上的解释结果集合。
   - `sweeps` - 所有 coverage lane 上生成出的 sweep 集合。
   - `sweep_graph_info` - cadence 求解直接消费的 sweep 图对象。
   - `sweep_cadence_info` - 基于 sweep graph 求出的最终 sweep 顺序骨架。
   - `final_coverage_path_info` - 最终 sweep 段与路口连接段已经物化后的路径结果。
4. 当前 `connection_unit` 的真实落点，是 `sweep_graph_build_info.sweep_graph_info.connection_units`
   - 它属于 `sweep_graph_info` 内部正式字段
   - 不是另起一层一级结果对象
5. 语义收口的目标，是把这条真实对象链讲清楚，而不是把源码尚未实现的另一套对象树硬写进文档。
