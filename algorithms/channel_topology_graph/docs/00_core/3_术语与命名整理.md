# 术语与命名整理

## 1. 文档目的

本文档只做术语收口，不改算法逻辑。

当前术语必须以现有源码对象为准，尤其是第 4 步对象名不能再凭文档重命名。

## 2. 阶段结果术语

- `GeometryPreparationResult`
  - 第 1 步结果对象
- `JunctionRebuildResult`
  - 第 2 步结果对象
- `TopologyGraphBuildResult`
  - 第 3 步结果对象
- `CoveragePlanningResult`
  - 第 4 步结果对象

## 3. 第 2/3 步静态对象术语

- `node_info`
  - 主拓扑层节点对象
- `edge_info`
  - 主拓扑层边对象
- `graph_info`
  - 由 `node_info + edge_info` 聚合出的正式图对象

## 4. 第 3 步派生图层术语

- `incident_port_info`
  - 节点局部 incident port 派生层
- `node_local_connection_hypothesis_info`
  - 节点局部连接假设派生层

这些都是第 3 步的正式 topology 派生信息，不再使用旧的 connection candidate / directed lane 术语。

## 5. 第 4 步边级覆盖术语

- `coverage_lane_info`
  - 边级覆盖解释层结果

当前它回答：

- 某条 edge 如何被覆盖
- 哪些区域归它负责
- 能否在其上生成 sweeps

## 6. 第 4 步 sweep 级静态图层术语

- `sweeps`
  - sweep 列表
- `sweep_group_info`
  - 按 coverage lane 聚合后的组信息
- `sweep_port_view_info`
  - 节点口排序视图
- `sweep_transition_candidate_info`
  - sweep 转移候选粗筛结果
- `sweep_graph_info`
  - 进入 cadence 求解的 sweep 图

这里要特别区分：

- `sweep_graph_info`
  - 是当前源码真实存在的对象
- `connection_unit`
  - 当前源码真实存在，但它落在 `sweep_graph_info.connection_units`
  - 不能在文档里把它偷拆成独立一级 `connection_unit_info`

## 7. 第 4 步结果层术语

- `sweep_cadence_info`
  - 最终 sweep 顺序骨架
- `final_coverage_path_info`
  - 最终离散路径结果

## 8. cadence route 术语

当前源码里 cadence route 的关键字段是：

- `sweep_sequence`
- `transition_sequence`
- `segments`
- `start_end_type`
- `end_end_type`

这套字段共同表达 route 骨架。

因此当前不应把 cadence 主序列文档化成源码里还没有的：

- `ordered_items`

## 9. final path 术语

当前 `final_coverage_path_info` 里使用的关键术语是：

- `routes`
- `ordered_items`
- `sweep_segments`
- `junction_connections`

其中：

- `ordered_items`
  - 是 final path 层自己的结果序列
- 不是 cadence 层当前源码 contract

## 10. 过程概念术语

以下术语可以继续存在，但只能作为过程概念：

- anchor
- sector
- truncation point
- fallback
- local feasible region

它们只能属于：

- 算法内部临时状态
- `debug_info`
- `validation_info`

## 11. 命名原则

1. 术语必须优先贴源码真实对象名
2. 语义收口时，允许补中文解释，不允许偷换对象名
3. 伪代码里的对象名，应尽量与源码函数和返回值一致
4. 若某个新术语只是解释层概念，必须明确写成“语义解释”，不能冒充当前 contract

## 12. 总结

当前术语收口应理解成：

- 第 1~4 步的阶段结果对象不变
- 第 4 步内部真实对象链不变
- 文档可以把这些对象按语义分层讲清楚
- 但不能把源码对象偷偷换成另一套名字
