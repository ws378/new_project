# 核心对象契约整理

## 1. 文档目标

本文档定义当前已经对齐、后续实现应直接遵循的正式 contract。

若源码尚未完全迁移到本文档的最新表述，以本文档作为实现目标和收口基线，不再继续保留旧主线语义。

本文档回答四类问题：

1. 四步阶段结果对象是什么
2. 第 2 步需要稳定哪些静态对象
3. 第 3 步的新主线契约是什么
4. 第 4 步覆盖规划模块内部应以什么对象链继续展开

本文档已经清理掉不再适合作为主线 contract 的旧内容。

这些旧派生层对象已经从当前正式 contract 中移除，不再作为主线结果对象或文档术语继续存在。

## 2. 当前正式阶段结果对象

当前四步阶段结果对象统一为：

1. `GeometryPreparationResult`
2. `JunctionRebuildResult`
3. `TopologyGraphBuildResult`
4. `CoveragePlanningResult`

这是当前主线的一级阶段结果对象，不再另发明并列一级对象体系。

---

## 3. `GeometryPreparationResult`

### 3.1 含义

- 第 1 步正式输出
- 承载运行尺度下的几何世界真值
- 是后续全部骨架、拓扑、coverage 判断的唯一几何输入起点

### 3.2 核心字段

- `gray` - 运行尺度下的原始灰度图，是后续几何与骨架处理的基础底图。
- `region_mask` - 参与本次规划的区域约束掩膜，限定算法有效工作范围。
- `free_mask` - 运行尺度下的自由空间掩膜，是后续通行与覆盖判断的基础真值。
- `obstacle_mask` - 运行尺度下的障碍物掩膜，用于和自由空间形成互补几何世界。
- `after_open_mask` - 形态学开操作后的自由区结果，用于稳定后续骨架与拓扑提取。
- `skeleton_mask` - 原始骨架掩膜，记录自由区中心线的初始提取结果。
- `skeleton_pruned_mask` - 裁剪和修枝后的骨架掩膜，是后续重建更稳定的骨架输入。
- `skeleton_pixels_rc` - 修枝后骨架像素坐标集合，便于后续按离散点消费骨架。
- `crop_box_px` - 本次处理区域在原图中的裁剪框，记录运行尺度与原图的映射范围。
- `resolution_m_per_px` - 像素到米制的分辨率，是全部长度换算的统一尺度基准。

### 3.3 最小闭环约束

1. `resolution_m_per_px > 0`
2. `free_mask / obstacle_mask` 与运行尺度一致
3. 后续所有空间合法性判断只信这一阶段输出的正式几何真值

## 4. 第 2 步静态对象契约

第 2 步交汇重建模块的职责是：

- 输出正式 `node_info / edge_info`
- 在节点侧补齐普通节点、`pure_cycle_cut`、`dead_end` 都可兼容的统一语义字段
- 不在这一阶段提前发明 node-local hypothesis 或 sweep 级对象

### 4.1 `NodeInfo`

#### 4.1.1 含义

- 第 2 步开始建立的正式节点对象
- 是后续 topology 与 coverage 消费的唯一节点静态真值

#### 4.1.2 核心字段

- `node_id` - 当前条目关联的正式节点 ID。
- `point_rc` - 节点在运行尺度下的中心坐标。
- `incident_edge_ids` - 与当前节点相连的正式边 ID 列表。
- `degree` - 当前节点的 incident edge 数量摘要。
- `polygon_vertices_rc` - 节点区域多边形顶点坐标，用于节点口与局部几何裁剪。
- `node_type` - 节点主分类语义，用于表达节点的基础类别，不承担全部特殊语义。
- `is_virtual` - 当前节点是否属于人工引入的虚拟语义节点。
- `virtual_reason` - 若 `is_virtual = true`，记录该节点被引入的原因。

#### 4.1.3 `node_type / is_virtual / virtual_reason` 的职责边界

- `node_type` - 保留现有主分类语义，负责表达节点的基础类别。
- `is_virtual` - 只回答“该节点是否是人工引入的语义节点”。
- `virtual_reason` - 只回答“为什么引入这个虚拟语义”。

当前至少需要兼容下面三类节点：

1. 普通节点
   - `is_virtual = false`
   - `virtual_reason = null`
2. `pure_cycle_cut` 节点
   - `is_virtual = true`
   - `virtual_reason = pure_cycle_cut`
3. `dead_end` 节点
   - `is_virtual = true`
   - `virtual_reason = dead_end`

#### 4.1.4 最小闭环约束

1. `node_id` 全局唯一
2. `degree == len(incident_edge_ids)`
3. `point_rc` 使用运行尺度坐标
4. 普通节点、`pure_cycle_cut`、`dead_end` 三类节点共用同一 `NodeInfo` 契约，不允许分裂成并列节点对象体系

### 4.2 `EdgeInfo`

#### 4.2.1 含义

- 第 2 步开始建立的正式边对象
- 是后续 topology、coverage 与 final path 继续消费的唯一边静态真值

#### 4.2.2 核心字段

- `edge_id` - 边的全局唯一标识，用于边级真值回指和后续覆盖映射。
- `src_node_id` - 当前边起点节点 ID，用于建立图上的有端点边关系。
- `dst_node_id` - 当前边终点节点 ID，用于建立图上的有端点边关系。
- `edge_type` - 当前边在端点结构上的基础类别，用于描述 normal / dead-end / loop 等边形态。
- `inner_path_rc` - 边的内侧路径点列，通常用于节点内局部几何参考。
- `outer_path_rc` - 边的外侧主路径点列，是 coverage lane 与 sweep 生成的重要几何主轴。
- `path_rc` - 当前边的完整正式点列坐标。
- `length_px` - 边路径在像素尺度下的长度。
- `length_m` - 边路径在米制尺度下的长度。
- `coverage_info` - 挂回到正式边对象上的边级 coverage 投影真值。

#### 4.2.3 `edge_type` 的职责边界

`edge_type` 只表达边在端点结构上的基础形态，不直接承担 topology hypothesis 或 coverage 动作语义。

当前建议至少支持：

- `normal_link` - 双端点正常连通边
- `single_dead_end_link` - 单端 dead-end 边
- `double_dead_end_link` - 双端 dead-end 边
- `cycle_loop` - 回环 / self-loop 边

#### 4.2.4 关于 edge 端点局部几何缓存

当前已经能通过 topology graph build 的派生视图恢复：

- `src.contact_rc`
- `dst.contact_rc`
- `src.heading_deg_image`
- `dst.heading_deg_image`

这些字段的含义是：

- `contact_rc` - 边在节点局部区域真正接入/接出的口点坐标。
- `heading_deg_image` - 边在该节点口从节点朝边主体离开的局部方向角。

当前阶段的契约选择是：

- 暂不把这些字段直接塞回 `EdgeInfo` 主体
- 它们继续作为 topology graph build 的派生视图存在
- 后续由 `incident_port_info` 消费

#### 4.2.5 `coverage_info` 当前核心字段

- `coverage_lane_id` - coverage lane 的唯一标识，用于 lane 与 sweep 的归属关联。
- `active` - 当前对象是否处于正式生效状态，用于区分有效与排除结果。
- `main_direction` - 当前 coverage lane 的主覆盖方向语义。
- `territory_pixels` - 当前 lane 或边所负责的覆盖责任区域像素集合。
- `effective_region_pixels` - 实际允许 sweep 落位的有效区域像素集合。
- `sweep_ids` - 当前 coverage lane 下挂接的正式 sweep ID 列表。
- `sweep_count` - 当前对象关联的正式 sweep 数量摘要。
- `local_width_stats` - 当前 lane 局部宽度相关的统计结果，用于覆盖宽度与布局判断。
- `geometry_valid` - 当前 lane 的几何合法性判定结果。
- `node_valid` - 当前 lane 与端点节点关系是否合法的判定结果。
- `topology_valid` - 当前 lane 在拓扑约束下是否成立的判定结果。
- `excluded_reason` - 当前对象被排除或失效时记录的明确原因。
- `resolution_m_per_px` - 像素到米制的分辨率，是全部长度换算的统一尺度基准。

#### 4.2.6 最小闭环约束

1. `edge_id` 全局唯一
2. `src_node_id / dst_node_id` 能回指正式 `NodeInfo`
3. `src_node_id == dst_node_id` 在契约上明确允许，不能再被视为非法边形态
4. `outer_path_rc` 是第 4 步 coverage lane 与 sweep 生成的关键几何输入
5. 第 4 步结果里的 `coverage_info.sweep_ids` 必须能回指当前 stage A sweeps
6. `coverage_info.territory_pixels / effective_region_pixels` 是当前 edge 的边级 coverage 区域真值
7. `local_width_stats` 是固定字段集合，不再允许以“任意附加键值”的裸字典持续扩张

## 5. `GraphInfo`

### 5.1 含义

- 第 3 步第一次建立的正式图对象
- 承载正式节点、正式边及其静态拓扑几何真值

### 5.2 核心字段

- `nodes: tuple[NodeInfo, ...]` - 正式图中的节点对象集合。
- `edges: tuple[EdgeInfo, ...]` - 正式图中的边对象集合。
- `meta` - 当前结果对象的附加元信息，用于记录运行上下文、阶段摘要、兼容期派生视图、调试留痕和不适合进入主体契约的辅助信息。

### 5.3 最小闭环约束

1. `nodes` 中每个元素都已 compact 完成
2. `edges` 中每个元素都能通过 `src_node_id / dst_node_id` 回指 `nodes`
3. 第 4 步结果中的 `graph_info.edges[*].coverage_info` 可以承接边级 coverage 投影
4. `GraphInfo` 仍不直接承接 sweeps、cadence 或 final path

## 6. `JunctionRebuildResult`

### 6.1 含义

- 第 2 步正式结果对象

### 6.2 核心字段

- `node_info_list` - junction_rebuild 阶段输出的正式节点列表。
- `edge_info_list` - junction_rebuild 阶段输出的正式边列表。
- `debug_info` - 面向排查和可视化核对的调试信息集合，用于保留中间过程、对比材料、渲染辅助数据和人工复核所需线索。
- `validation_info` - 面向契约闭环检查的校验结果集合，用于记录主键回指、几何合法性、拓扑闭环和阶段内一致性检查结果。
- `meta` - 当前结果对象的附加元信息，用于记录运行上下文、阶段摘要、兼容期派生视图、调试留痕和不适合进入主体契约的辅助信息。

### 6.3 最小闭环约束

1. 只输出正式 `node_info / edge_info`
2. 不在这里偷带 `graph_info`
3. 不在这一阶段提前混入 `incident_port_info` 或 hypothesis 对象

## 7. 第 3 步主线契约

第 3 步拓扑图构建模块的职责已重定义为：

1. 建立正式 `graph_info`
2. 基于 `graph_info` 构造 `incident_port_info`
3. 基于 port 真值构造 `node_local_connection_hypothesis_info`

旧的 topology 中间派生层已经退出正式主线契约，不再在本文档中作为主线 contract 继续描述。

### 7.1 `TopologyGraphBuildResult`

#### 7.1.1 含义

- 第 3 步正式结果对象

#### 7.1.2 核心字段

- `graph_info` - 当前阶段使用或产出的正式图对象，是节点、边及其静态拓扑几何真值的统一承载体，也是后续 coverage 继续复用的基础输入。
- `incident_port_info` - 第 3 步构造的节点局部 incident port 真值集合，是 node-local hypothesis 的直接输入。
- `node_local_connection_hypothesis_info` - 第 3 步构造的 node-local connection hypothesis 真值集合，是第 4 步继续展开 sweep transition candidates 的直接输入。
- `debug_info` - 第 3 步整阶段的调试留痕容器，用于保留阶段性中间结构、对比材料和人工审查线索。
- `validation_info` - 第 3 步整阶段的校验结果容器，用于记录对象回指、port 排序合法性、hypothesis 闭环和阶段一致性检查结果。
- `meta` - 第 3 步整阶段的附加元信息容器，用于记录运行上下文、计数摘要和兼容期辅助视图。

#### 7.1.3 最小闭环约束

1. 第 3 步第一次正式建立 `graph_info`
2. `incident_port_info` 必须建立在 `graph_info` 的正式节点和边之上
3. `node_local_connection_hypothesis_info` 必须建立在 `incident_port_info` 之上
4. 第 4 步必须继续复用这个 `graph_info`
5. 第 4 步 sweep transition expansion 直接读取 `node_local_connection_hypothesis_info`

### 7.2 `incident_port_info`

#### 7.2.1 含义

- 第 3 步第一层派生结果
- 表达某个 node 上，哪些 edge 端点出现构成了可供枚举的局部 port
- 它是正式主线输入，不再只是 `graph_info.meta` 里的内部缓存

#### 7.2.2 核心字段

- `items` - 当前对象中的全部 port 条目列表，是主列表承载字段。
- `items_by_node` - 按 `via_node_id` 聚合的 port 条目视图，便于按节点局部消费。
- `summary` - port 规模和统计摘要。

每个 port 至少包含：

- `port_id` - port 的唯一标识。
- `via_node_id` - 当前 port 所属节点 ID。
- `edge_id` - 当前 port 所属正式边 ID。
- `end_type` - 当前 port 对应边的哪一端，显式取值 `src / dst`。
- `peer_node_id` - 当前 port 对侧关联的节点 ID；若为 same-node edge，则允许与 `via_node_id` 相同。
- `contact_rc` - 当前 port 在节点局部区域的接入/接出口点坐标。
- `heading_deg_image` - 当前 port 从节点朝边主体离开的局部方向角。
- `cw_order_index` - 当前 port 在节点局部顺时针序中的位置编号。
- `cw_prev_port_id` - 当前 port 在节点局部顺时针序中的前一个 port。
- `cw_next_port_id` - 当前 port 在节点局部顺时针序中的后一个 port。

#### 7.2.3 最小闭环约束

1. `port_id` 在当前 stage 内唯一
2. `via_node_id` 能回指正式 `NodeInfo`
3. `edge_id` 能回指正式 `EdgeInfo`
4. `end_type` 只能取 `src / dst`
5. `cw_prev_port_id / cw_next_port_id` 必须能回指当前节点局部 port 集
6. port 的顺时针排序单位是 port，不再是 edge

### 7.3 `node_local_connection_hypothesis_info`

#### 7.3.1 含义

- 第 3 步第二层派生结果
- 表达某个 node 上，哪些 `in_port -> out_port` 关系在 node-local 语义上成立
- 它是 topology 层对象，不是 sweep 层对象

#### 7.3.2 核心字段

- `items` - 当前对象中的全部 hypothesis 条目列表，是主列表承载字段。
- `items_by_node` - 按 `via_node_id` 聚合的 hypothesis 视图，便于按节点局部排查。
- `summary` - hypothesis 规模和统计摘要。

每个 hypothesis 至少包含：

- `hypothesis_id` - hypothesis 的唯一标识。
- `via_node_id` - 当前 hypothesis 所属节点 ID。
- `in_port_id` - 入侧 port ID。
- `out_port_id` - 出侧 port ID。
- `in_edge_id` - 入侧正式边 ID。
- `in_end_type` - 入侧边端点类型真值。
- `out_edge_id` - 出侧正式边 ID。
- `out_end_type` - 出侧边端点类型真值。
- `connection_kind` - 当前 hypothesis 的连接语义类型。
- `base_confidence` - 当前 hypothesis 在 topology 层的基础置信度。
- `reason_tags` - 当前 hypothesis 生成原因和解释标签。

#### 7.3.3 `connection_kind` 当前至少支持

- `through`
- `turn`
- `cycle_through`
- `dead_end_return`

#### 7.3.4 最小闭环约束

1. `hypothesis_id` 在当前 stage 内唯一
2. `in_port_id / out_port_id` 必须能回指 `incident_port_info.items`
3. `in_end_type / out_end_type` 必须直接来自 port 真值，不允许下游再猜
4. `node_local_connection_hypothesis_info` 不再输出 `keep / reject / defer` 旧三分类主线结果
5. 当前对象是第 4 步 `sweep_transition_candidate_info` 的直接上游输入

## 8. 第 4 步主线契约

第 4 步 coverage planning 模块整体仍然保留，但其内部主线已收敛为：

1. `coverage lane + sweep generation`
2. `sweep transition expansion + score`
3. `cadence action selection`
4. `final path materialization`

也就是说：

- coverage planning 不是只剩后三块
- 而是原模块内部与 sweep 转移、cadence、final path 有关的子模块，将被这三条新主线吸收替换

同时需要明确：

- `sweep_transition_candidate_info` 继续保留
- `sweep_graph_info` 不再作为长期正式主线对象继续保留
- 它当前承接的功能，应拆回 `coverage_lane_sweep_info`、`graph_info` 和后续规范化后的 sweep transition 真值，而不是继续维持一个名字过大的中间壳对象

### 8.1 `CoveragePlanningResult`

#### 8.1.1 含义

- 第 4 步正式结果对象

#### 8.1.2 当前正式字段

- `graph_info` - 第 4 步继续沿用第 3 步建立的正式图对象，承载节点、边、coverage 回写入口和全部静态拓扑几何真值。
- `coverage_lane_sweep_info` - 第 4 步前半段的正式 build-info，用于收口边级 coverage lane 真值与 sweeps 真值，是 coverage 几何铺设层的统一输出对象。
- `sweep_transition_candidate_info` - 第 4 步 transition 扩展与打分阶段的正式 candidate 对象，用于承接 node-local hypothesis 到 sweep candidate 的展开结果。
- `sweep_cadence_build_info` - 第 4 步 cadence 子域的正式 build-info，用于收口最终 sweep 节拍骨架、route 结果和覆盖统计，是 coverage 时序求解层的正式输出对象。
- `final_coverage_path_build_info` - 第 4 步 final path 子域的正式 build-info，用于收口最终路径物化真值、路口连接结果和最终校验结果，是 coverage 几何落成层的正式输出对象。
- `debug_info` - 面向排查和可视化核对的调试信息集合，用于保留中间过程、对比材料、渲染辅助数据和人工复核所需线索。
- `validation_info` - 面向契约闭环检查的校验结果集合，用于记录主键回指、几何合法性、拓扑闭环和阶段内一致性检查结果。
- `meta` - 当前结果对象的附加元信息，用于记录运行上下文、阶段摘要、兼容期派生视图、调试留痕和不适合进入主体契约的辅助信息。

#### 8.1.3 顶层通用字段补充说明

- `graph_info` - 第 4 步继续沿用第 3 步建立的正式图对象，承载节点、边、coverage 回写入口和全部静态拓扑几何真值。
- `debug_info` - 第 4 步整阶段的调试留痕容器，用于保留中间统计、阶段性对比材料、渲染辅助数据、异常现场和人工复核所需线索。
- `validation_info` - 第 4 步整阶段的校验结果容器，用于记录对象闭环、主键回指、阶段间一致性、路径合法性和失败原因摘要。
- `meta` - 第 4 步整阶段的附加元信息容器，用于记录运行上下文、摘要计数、兼容期镜像视图和不适合写入主体契约的辅助信息。

#### 8.1.4 四个 build-info / candidate 字段的职责

- `coverage_lane_sweep_info` - coverage 几何铺设层的正式 build-info，对边级 coverage lane 和 sweep 真值做统一收口。
- `sweep_transition_candidate_info` - sweep transition expansion + score 的正式结果对象，对 sweep-to-sweep candidates、分档和评分真值做统一收口。
- `sweep_cadence_build_info` - sweep 时序求解阶段的正式 build-info，对 cadence routes、route 顺序骨架和覆盖统计做统一收口。
- `final_coverage_path_build_info` - 最终路径物化阶段的正式 build-info，对最终路径、路口连接结果和最终校验做统一收口。

#### 8.1.5 语义分层

这些字段当前按正式主装配收成四层：

1. `coverage_lane_sweep_info`
   - `coverage_lane_info`
   - `sweeps`
2. `sweep_transition_candidate_info`
   - `items`
   - `summary`
3. `sweep_cadence_build_info`
   - `sweep_cadence_info`
   - `coverage_stats`
4. `final_coverage_path_build_info`
   - `final_coverage_path_info`

#### 8.1.6 最小闭环约束

1. `graph_info` 继续引用第 3 步建立的正式图对象
2. `coverage_lane_sweep_info` 必须完整承载 `coverage_lane_info + sweeps`
3. `sweep_transition_candidate_info` 必须建立在 `node_local_connection_hypothesis_info` 之上
4. `sweep_cadence_build_info` 不应再以 `sweep_graph_info` 作为长期正式主线输入；其真实承接功能应拆回 `coverage_lane_sweep_info`、`graph_info` 和经规范化后的 sweep transition 真值
5. `final_coverage_path_build_info` 必须建立在 `coverage_lane_sweep_info + sweep_cadence_build_info` 之上
6. `CoveragePlanningResult` 不再保留旧的 sweep transition candidate / directed lane 顶层镜像字段

### 8.2 `coverage_lane_info`

#### 8.2.1 含义

- 第 4 步前半段输出的边级覆盖解释层
- 当前仍作为阶段对象保留，用于 compare、阶段 JSON 落盘和局部审查
- 长期其边级真值仍应逐步收回 `EdgeInfo.coverage_info`

#### 8.2.2 当前核心字段

- `coverage_lane_id` - coverage lane 的唯一标识，用于 lane 与 sweep 的归属关联。
- `source_edge_id` - 当前 lane 所来源的正式边 ID。
- `main_direction` - 当前 coverage lane 的主覆盖方向语义。
- `territory_pixels` - 当前 lane 所负责的覆盖责任区域像素集合。
- `effective_region_pixels` - 实际允许 sweep 落位的有效区域像素集合。
- `sweep_ids` - 当前 coverage lane 下挂接的正式 sweep ID 列表。
- `sweep_count` - 当前对象关联的正式 sweep 数量摘要。
- `local_width_stats` - 当前 lane 局部宽度相关的统计结果，用于覆盖宽度与布局判断。
- `geometry_valid` - 当前 lane 的几何合法性判定结果。
- `node_valid` - 当前 lane 与端点节点关系是否合法的判定结果。
- `topology_valid` - 当前 lane 在拓扑约束下是否成立的判定结果。
- `active` - 当前对象是否处于正式生效状态，用于区分有效与排除结果。
- `excluded_reason` - 当前对象被排除或失效时记录的明确原因。

#### 8.2.3 最小闭环约束

1. `coverage_lane_id` 在当前 stage 内唯一
2. `source_edge_id` 能回指正式 `EdgeInfo`
3. `sweep_ids` 中每个元素都能回指 `sweeps`
4. `local_width_stats` 只承载 sweep 铺设宽度统计，不允许混入无关 debug 信息

### 8.3 `sweeps`

#### 8.3.1 含义

- 第 4 步前半段生成的正式 sweep 列表

#### 8.3.2 当前核心字段

- `sweep_id` - sweep 的唯一标识，用于 cadence 与 final path 回指。
- `coverage_lane_id` - coverage lane 的唯一标识，用于 lane 与 sweep 的归属关联。
- `source_edge_id` - 当前 sweep 所来源的正式边 ID。
- `path_rc` - 当前 sweep 的正式点列坐标。
- `anchor_points_rc` - 当前 sweep 的锚点坐标序列，用于表达离散布局参考位置。
- `offset_profile_px` - 当前 sweep 的横向偏移剖面真值，用于表达布局偏移分布。
- `path_length_px` - 当前 sweep 在像素尺度下的长度。
- `path_length_m` - 当前 sweep 在米制尺度下的长度。
- `side_label` - 当前 sweep 相对 lane 横向位置的语义标签。
- `side_level` - 当前 sweep 在横向序列中的层级或排序级别。
- `active` - 当前对象是否处于正式生效状态，用于区分有效与排除结果。

#### 8.3.3 最小闭环约束

1. `sweep_id` 在当前 stage 内唯一
2. `coverage_lane_id` 能回指 `coverage_lane_info`
3. `path_rc` 至少包含 2 个点
4. `offset_profile_px` 的长度应与当前 sweep 的锚点采样数一致

### 8.4 `sweep_transition_candidate_info`

#### 8.4.1 含义

- 第 4 步 `sweep transition expansion + score` 的正式结果对象
- 用于把 topology 层的 `node_local_connection_hypothesis_info` 展开成可进入 cadence 的 sweep-to-sweep 正式候选，并保留端点几何、局部可行性、风险与收益评分

#### 8.4.2 核心字段

- `items` - 当前对象中的正式 sweep transition candidate 条目列表，是 cadence 唯一读取的候选主列表。
- `summary` - candidate 规模和统计摘要。

每个 candidate 至少包含：

- `candidate_id` - 当前 sweep candidate 的唯一标识。
- `source_hypothesis_id` - 所来源的 topology hypothesis ID。
- `via_node_id` - 当前 candidate 所属节点 ID。
- `from_sweep_id` - 起始 sweep ID。
- `to_sweep_id` - 目标 sweep ID。
- `from_end_type` - 起始 sweep 的端点类型真值。
- `to_end_type` - 目标 sweep 的端点类型真值。
- `connection_kind` - 直接继承自 topology hypothesis 的连接语义。
- `mapping_type` - 当前 sweep pair 的形成方式标签，用于解释候选来自跨 group 几何配对、同 group 横移还是 foldback。
- `base_confidence` - 继承自 topology hypothesis 的基础置信度。
- `endpoint_distance_px` - from sweep 退出端与 to sweep 进入端之间的像素距离，是跨 group 候选排序和过滤的重要几何量。
- `sweep_turn_delta_deg` - 由 from sweep 退出端 `A/B` 与 to sweep 进入端 `C/D` 计算得到的 sweep 级端点方向夹角，用于校验 topology 层 `motion_type` 是否落到具体 sweep pair 上仍然合理。
- `local_feasibility_score` - 当前 pair 在局部可行域、障碍风险和连接空间中的可行性评分。
- `rank_gap` - 同 group 横移候选的端口序差；当前未建立跨 group rank frame 对齐，跨 group 候选只能把它作为解释字段。
- `risk_score` - 当前 candidate 的风险分数。
- `coverage_gain_score` - 当前 candidate 的覆盖收益分数。
- `total_score` - 当前 candidate 的综合评分。
- `reason_tags` - 当前 candidate 的解释标签集合。

#### 8.4.3 最小闭环约束

1. `source_hypothesis_id` 必须能回指 `node_local_connection_hypothesis_info.items`
2. `from_end_type / to_end_type` 必须由算法阶段真值直接给出
3. `connection_kind` 必须从 topology hypothesis 直接继承，不允许在这一层被压平成普通 through
4. 跨 group candidate 必须由 topology 许可、sweep 端点距离、sweep 级转角和局部可行性共同决定
5. 跨 group candidate 不能由 `rank_gap / mean_offset_m / side_level` 主导排序或过滤
6. group 内 candidate 可以使用 `rank_gap / mean_offset_m / side_level`，但必须处在同一 group frame 内，并结合端点几何
7. cadence 只能读取 `items`，不能读取另设的 kept / weak / fallback 平行集合

### 8.5 `sweep_cadence_info`

#### 8.5.1 含义

- 基于 sweep transition candidates 继续求出的最终 sweep 顺序骨架
- 是 `cadence action selection` 的正式结果对象

#### 8.5.2 当前核心字段

- `routes` - cadence 求解得到的全部正式 route 列表。
- `route_sweep_order` - 各条 route 内 sweep 访问顺序的全局展开视图。
- `ordered_passes` - 将 route 展开成 sweep 与 connection/u-turn 经过顺序的显式时间线视图。
- `summary` - 当前对象的规模与摘要信息。

每条 route 至少包含：

- `route_id` - 当前 route 的唯一标识。
- `sweep_sequence` - 当前 route 中按访问顺序排列的 sweep ID 序列。
- `transition_sequence` - 当前 route 中按经过顺序记录的 transition candidate 或后续规范化 transition 真值 ID 序列。
- `segments` - 当前 route 中 sweep 之间连接原语的压缩段序列。
- `ordered_passes` - 将 route 展开成 sweep 与 connection/u-turn 经过顺序的显式时间线视图。
- `start_sweep_id` - 当前 route 的起始 sweep ID。
- `end_sweep_id` - 当前 route 的结束 sweep ID。
- `start_end_type` - 当前 route 起始进入端型。
- `end_end_type` - 当前 route 结束离开端型。
- `sweep_count` - 当前 route 覆盖的 sweep 数量。
- `transition_count` - 当前 route 经过的 transition 数量。
- `segment_count` - 当前 route 中 segment 的数量。
- `transition_cost` - 当前 route 中 transition 部分的代价统计。
- `connector_cost` - 当前 route 中连接器相关的代价统计。

#### 8.5.3 `segments` 语义要求

每个 segment 至少还应满足：

- `primitive_type` - 当前动作原语类型，至少区分 `transition / u_turn`。
- `connection_kind` - 若 `primitive_type = transition`，必须显式区分 `through / turn / cycle_through / dead_end_return`。
- `transition_action_type` - 当前 transition 动作在 cadence 动作层的正式类型；其中 `dead_end_return` 必须作为正式 transition action 进入 cadence，而不是继续退化为 fallback `u_turn`。

#### 8.5.4 最小闭环约束

1. `sweep_sequence` 与 `transition_sequence / segments` 连续
2. 全部 sweeps 最终都应被 cadence 覆盖
3. `ordered_passes` 必须与 `segments` 一一对应展开为 `sweep -> connection/u_turn -> sweep`
4. `segments[*].entry_end_type / exit_end_type` 只服务于 cadence 连续性与 `u_turn` 原语，不作为普通 connection 的正式端点真值来源
5. `segments[*].connection_kind = dead_end_return` 时，必须被视为正式 transition action，而不是 fallback `u_turn`

### 8.6 `coverage_stats`

#### 8.6.1 含义

- 第 4 步 cadence 子域的覆盖统计结果

#### 8.6.2 当前核心字段

- `total_sweep_count` - 正式 sweep 全集的总数量。
- `covered_sweep_count` - 已被 cadence 正式覆盖到的 sweep 数量。
- `uncovered_sweep_count` - 仍未被 cadence 覆盖到的 sweep 数量。
- `coverage_ratio` - 已覆盖 sweep 数占 sweep 全集的比例。
- `is_complete` - 当前 cadence 是否已经完整覆盖 sweep 全集。

#### 8.6.3 最小闭环约束

1. 只允许由 `coverage_lane_sweep_info.sweeps` 与 `sweep_cadence_build_info.sweep_cadence_info` 推导
2. `covered_sweep_count <= total_sweep_count`
3. `coverage_ratio == covered_sweep_count / total_sweep_count`
4. `is_complete == (covered_sweep_count == total_sweep_count)`

### 8.7 `FinalCoveragePathOrderedItem`

#### 8.7.1 含义

- final path route 内部的结果 item

#### 8.7.2 当前 item 类型

1. `sweep_segment`
2. `junction_connection`

#### 8.7.3 关键字段

- `item_type` - final path 有序项的类型标记。
- `route_id` - 当前 route 的唯一标识。
- `item_index` - 当前 item 在 route 内的有序位置。
- `sweep_id` - 当前 sweep 的唯一标识，用于 cadence 与 final path 回指。
- `connection_id` - 路口连接的全局唯一标识。
- `path_points_rc` - 当前 item 对应的正式路径点列。
- `sweep_points_rc` - 当前 sweep 段在最终路径中的点列。
- `junction_connection_points_rc` - 当前路口连接段的点列。

### 8.8 `FinalCoveragePathConnection`

#### 8.8.1 含义

- final path 中单条路口连接结果

#### 8.8.2 当前核心字段

- `connection_id` - 路口连接的全局唯一标识。
- `route_id` - 当前 route 的唯一标识。
- `from_sweep_id` - 当前连接的起始 sweep ID。
- `to_sweep_id` - 当前连接的目标 sweep ID。
- `via_node_id` - 当前连接经过的节点 ID。
- `connection_class` - 当前连接的类别语义。
- `connector_kind` - 当前 connector 的正式语义类型，至少区分 `transition / cycle_through / dead_end_return / fallback_u_turn`。
- `is_constructible` - 当前连接是否成功构造。
- `failure_reason` - 构造失败时记录失败原因。
- `point_a_rc` - 连接规则中的 A 点。
- `point_b_rc` - 连接规则中的 B 点。
- `point_c_rc` - 连接规则中的 C 点。
- `point_d_rc` - 连接规则中的 D 点。
- `path_points_rc` - 当前连接物化后的正式路径点列。
- `path_length_px` - 当前连接在像素尺度下的长度。
- `path_length_m` - 当前连接在米制尺度下的长度。
- `theta_deg` - 当前连接规则的主方向角。
- `coverage_support_width_m` - 当前连接为覆盖连续性提供的支持宽度。
- `is_u_turn` - 当前连接是否属于 U 型回转连接。

#### 8.8.3 最小闭环约束

1. 成功构造时 `path_points_rc` 至少 2 点
2. 成功构造时首点必须等于 `point_b_rc`
3. 成功构造时尾点必须等于 `point_c_rc`
4. 失败构造时 `failure_reason` 不能为空
5. `connector_kind = cycle_through` 不能被误并到 `fallback_u_turn`
6. `connector_kind = dead_end_return` 不能被误并到普通 `transition` through

### 8.9 `FinalCoveragePathRoute`

#### 8.9.1 含义

- final path 的 route 级结果

#### 8.9.2 当前核心字段

- `route_id` - 当前 route 的唯一标识。
- `ordered_items` - 当前 route 或最终结果按真实经过顺序平铺的有序项集合。
- `junction_connections` - 当前 route 或全局范围内的路口连接结果集合。
- `path_subchains_rc` - 当前 route 内最终可连续执行的路径子链集合。
- `path_point_count` - 当前路径对象的总点数。
- `path_length_px` - 当前路径对象在像素尺度下的长度。
- `path_length_m` - 当前路径对象在米制尺度下的长度。
- `coverage_support_info` - 当前 route 为覆盖支撑宽度与拼接关系保留的辅助信息。

#### 8.9.3 最小闭环约束

1. `ordered_items` 保持当前 route 的真实展开顺序
2. 同一条连续子链内部，相邻可构造 item 的 seam 必须连续
3. 失败 `junction_connection` 允许把 route 显式切成多条连续子链
4. `path_subchains_rc` 必须与 `ordered_items` 的真实拼接结果一致

### 8.10 `FinalCoveragePathInfo`

#### 8.10.1 含义

- 第 4 步最终路径结果对象

#### 8.10.2 当前核心字段

- `routes` - cadence 求解得到的全部正式 route 列表。
- `ordered_items` - 当前 route 或最终结果按真实经过顺序平铺的有序项集合。
- `junction_connections` - 当前 route 或全局范围内的路口连接结果集合。
- `summary` - 当前对象的规模与摘要信息。

#### 8.10.3 最小闭环约束

1. `junction_connections` 的 `connection_id` 全局唯一
2. route 级 `ordered_items` 与 `junction_connections / sweep_segments` 一致
3. `summary` 应与 routes 聚合结果一致

## 9. 总结

当前核心 contract 可以收成三层：

1. 阶段结果对象
   - `GeometryPreparationResult`
   - `JunctionRebuildResult`
   - `TopologyGraphBuildResult`
   - `CoveragePlanningResult`
2. 第 2/3 步静态与 topology 对象
   - `NodeInfo`
   - `EdgeInfo`
   - `GraphInfo`
   - `incident_port_info`
   - `node_local_connection_hypothesis_info`
3. 第 4 步 coverage 主线对象
   - `coverage_lane_info`
   - `sweeps`
   - `sweep_transition_candidate_info`
   - `sweep_cadence_info`
   - `final_coverage_path_info`

后续文档与实现若继续收语义，必须先以这套 contract 为准，不再回退到旧的 `connection_candidate / directed_lane_*` 主线表述。
