# 模块边界整理

## 1. 文档目的

本文档定义当前源码下的模块边界：

1. 各 stage 读什么、写什么
2. 第 4 步内部子模块怎么分
3. renderer 可以做什么，不可以做什么

## 2. 全局边界原则

## 2.1 pipeline 只管阶段串联

pipeline 层只负责：

- stage 顺序
- config 和 context 传递
- 结果汇总

不负责：

- 算法求解
- 审图真值生成

## 2.2 contracts 定义主线契约对象

contracts 层负责：

- stage result 对象
- `node_info / edge_info / graph_info`
- `SweepGraph* / SweepCadence* / FinalCoveragePath*` typed dict
- stage build-info 与对象级闭环契约字段

不负责：

- 具体求解规则
- 伪代码流程

## 2.3 algorithms 负责求解

algorithms 层负责：

- coverage lane 生成
- sweeps 生成
- sweep 图与 cadence
- final path 展开

算法层可以有内部中间量，但不能借文档整理之机被重新定义成另一套正式对象。

## 2.4 renderers 只读算法结果

renderers 只能读取：

- `CoveragePlanningResult`
  - 及其内部各对象

不能做：

- 再算一遍 cadence
- 再造一套 P/Q 或 A/B/C/D
- 再拼一条“更顺”的 final path

## 3. 四步 stage 边界

## 3.1 第 1 步

建议入口：

- `stages/geometry_preparation.py`

正式写出：

- `GeometryPreparationResult`

## 3.2 第 2 步

建议入口：

- `stages/junction_rebuild.py`

正式写出：

- `JunctionRebuildResult`

## 3.3 第 3 步

建议入口：

- `stages/topology_graph_build.py`

正式写出：

- `TopologyGraphBuildResult`

当前真实内容包括：

- `graph_info`
- `incident_port_info`
- `node_local_connection_hypothesis_info`

## 3.4 第 4 步

建议入口：

- `stages/coverage_planning.py`

正式写出：

- `CoveragePlanningResult`

## 4. 第 4 步内部边界

当前源码里第 4 步入口函数是：

- `build_coverage_plan(...)`

它内部按下面顺序组织：

1. `build_coverage_lane_sweep_info`
2. `attach_edge_coverage_info`
3. `build_sweep_graph_info`
4. `build_sweep_cadence_info`
5. `build_final_coverage_path_info`

因此第 4 步内部可以分成四层：

1. 边级覆盖解释层
2. sweep 静态与粗筛图层
3. cadence 层
4. final path 层

## 5. 第 4 步内部伪代码

```python
def build_coverage_plan(topology_graph_build_result, config, context):
    geometry_result = context["geometry_preparation_result"]

    coverage_lane_sweep_info = build_coverage_lane_sweep_info(...)
    graph_info = attach_edge_coverage_info(...)
    sweep_graph_build_info = build_sweep_graph_info(...)
    sweep_cadence_build_info = build_sweep_cadence_info(...)
    final_coverage_path_build_info = build_final_coverage_path_info(...)

    return CoveragePlanningResult(...)
```

## 6. 当前最该避免的越界行为

1. 第 3 步文档里提前写进 sweep 或 connection unit 正式对象
2. 第 4 步文档里把真实 `sweep_graph_info` 偷换成别的对象
3. renderer 自己重算 cadence 或 final path
4. final path 文档里忽略 `route_sweep_order / ordered_passes` 已经成为正式 cadence contract 字段

## 7. 总结

当前模块边界应理解成：

1. stage 之间靠阶段结果对象闭环
2. 第 4 步内部靠真实函数链闭环
3. 文档可以加强解释和注释，但不能改掉当前源码的模块分工
