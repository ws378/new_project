# 拓扑图构建模块方案

## 1. 文档定位

本文档描述当前第 3 步 `topology_graph_build` 模块的正式边界、正式主线和正式输出。

它必须与当前已经对齐的主线口径一致：

- 第 3 步第一次正式建立 `graph_info`
- 第 3 步的长期正式主线是 `graph_info + incident_port_info + node_local_connection_hypothesis_info`
- 本阶段的正式目标是“把第 2 步节点边结果收束成正式图，并建立 node-local 连接语义上游图层”

本文档不讨论第 4 步 sweep、cadence、final path 的算法，只把第 3 步自己的 contract 和主线说清楚。

同时要明确：pure cycle cut、dead_end 与 same-edge 真经过边的处理，属于第 3 步拓扑语义层职责，不应继续推给第 4 步临时兜底。

## 2. 当前输入与输出

输入：

- `JunctionRebuildResult`
  - `node_info_list` - 第 2 步输出的正式节点列表。
  - `edge_info_list` - 第 2 步输出的正式边列表。
  - `debug_info` - 第 2 步调试信息集合。
  - `validation_info` - 第 2 步校验结果集合。

输出：

- `TopologyGraphBuildResult`
  - `graph_info` - 第 3 步首次建立的正式图对象，是后续主线的图真值承载体。
  - `incident_port_info` - 在单节点局部展开的可连接 port 视图。
  - `node_local_connection_hypothesis_info` - 在单节点局部枚举的 `in_port -> out_port` 连接假设图层。
  - `debug_info` - 第 3 步调试信息集合。
  - `validation_info` - 第 3 步分层校验结果集合。
  - `meta` - 第 3 步规模摘要与阶段标记。

## 3. 本模块回答什么问题

当前第 3 步不是覆盖规划步骤，它回答的是：

1. 第 2 步输出的节点和边如何聚合成正式图对象。
2. pure cycle cut、dead_end 这些特殊拓扑语义如何在本阶段落定。
3. 节点局部可连接 port 视图如何建立。
4. 基于这个正式图对象，`in_port -> out_port` 连接假设如何枚举与分类。
5. 哪些 node-local 连接语义最终作为第 4 步 sweep transition expansion 的正式上游输入。

这一步不回答：

1. `coverage_lane_info` 是什么。
2. `sweeps` 是什么。
3. `sweep_transition_candidate_info` 如何评分。
4. `sweep_cadence_info` 怎么走。
5. `final_coverage_path_info` 怎么展开。

## 4. 当前真实主线

## 4.1 Stage 主装配主线

当前第 3 步 stage 入口可以按下面理解：

```python
def build_topology_graph(junction_rebuild_result, config=None, context=None):
    """第 3 步主入口。

    正式职责：
        1. 由 junction_rebuild 的正式 `node_info / edge_info` 第一次建立 `graph_info`
        2. 在 `graph_info` 上继续构造 `incident_port_info / node_local_connection_hypothesis_info`
        3. 汇总第 3 步的 validation / debug / meta，组装成正式 `TopologyGraphBuildResult`
    """

    # 第 3 步当前不消费 context。
    # 这里保留接口只是为了让 pipeline 编排层不必为本阶段写特殊分支。
    _ = context

    # 配置统一转成普通 dict，避免后续 helper 在 Mapping / 自定义对象上分叉。
    config = dict(config or {})

    # 第 3 步正式产物按固定顺序构建：
    # graph_info -> incident ports -> node-local connection hypotheses
    stage_outputs = build_topology_graph_stage_outputs(
        junction_rebuild_result=junction_rebuild_result,
        config=config,
    )

    # 校验也按同一层次结构分层返回，方便 baseline 直接定位偏差发生在哪一层。
    validation_info = build_topology_graph_validation_info(stage_outputs)

    # 最后统一装箱成正式 stage result，不在主入口继续散挂局部变量。
    return build_topology_graph_result(
        stage_outputs=stage_outputs,
        validation_info=validation_info,
    )
```

## 4.2 子模块展开主线

如果把第 3 步展开到子模块级别，真实主线可以写成：

```python
def build_topology_graph_stage_outputs(*, junction_rebuild_result, config):
    # 第 3 步第一次正式建图。
    graph_info = assemble_graph_info(
        node_info_list=junction_rebuild_result.node_info_list,
        edge_info_list=junction_rebuild_result.edge_info_list,
    )

    # 节点局部 incident port 视图是 node-local hypothesis 层的唯一正式入口。
    incident_port_info = build_incident_ports(graph_info)

    # 在单节点内部枚举 in_port -> out_port 候选，并做 connection_kind / confidence 分类。
    node_local_connection_hypothesis_info = build_node_local_connection_hypotheses(
        graph_info,
        incident_port_info,
        config,
    )

    return {
        "graph_info": graph_info,
        "incident_port_info": incident_port_info,
        "node_local_connection_hypothesis_info": node_local_connection_hypothesis_info,
    }
```

上面这段伪代码表达的是当前阶段职责，而不是后续 coverage 层逻辑。

## 4.3 `assemble_graph_info(...)` 伪代码

```python
def assemble_graph_info(node_info_list, edge_info_list):
    """把 junction_rebuild 输出聚合成正式图对象。"""

    # 第 3 步不新造 node / edge，只收 junction_rebuild 已经 compact 完成的正式输出。
    nodes = tuple(node_info_list)
    edges = tuple(edge_info_list)

    # graph_info 在这里第一次建立。
    # meta 只记录这是 topology_graph_build 首次建图，以及节点边规模。
    return GraphInfo(
        nodes=nodes,
        edges=edges,
        meta={
            "stage": "topology_graph_build",
            "node_count": len(nodes),
            "edge_count": len(edges),
        },
    )
```

这一步的长期约束很明确：

- 只聚合，不重算 node / edge 规则。
- 不在这里偷带 sweep、cadence、final path 对象。
- `graph_info` 从这里开始成为后续第 3、4 步共同消费的正式入口。

## 4.4 `build_incident_ports(...)` 伪代码

```python
def build_incident_ports(graph_info):
    """为每个节点建立顺时针 incident port 局部视图。"""

    # 先按 edge_id 预计算端点几何，避免在 src/dst 两端节点上重复恢复。
    endpoint_geometry_by_edge = {}
    for edge in graph_info.edges:
        endpoint_geometry_by_edge[edge.edge_id] = build_edge_endpoint_geometry(edge)

    port_items = []
    edge_by_id = {edge.edge_id: edge for edge in graph_info.edges}

    for node in graph_info.nodes:
        node_id = node.node_id
        local_ports = []

        # 对当前节点的每条 incident edge，恢复“接触点 / 切向 / 图像坐标系 heading / 对端节点”这些局部真值。
        for edge_id in node.incident_edge_ids:
            edge = edge_by_id[edge_id]
            endpoint_geometry = endpoint_geometry_by_edge[edge.edge_id]

            if edge.src_node_id == node_id:
                local_ports.append(
                    build_incident_port_item(
                        node_id=node_id,
                        edge=edge,
                        end_type="src",
                        peer_node_id=edge.dst_node_id,
                        endpoint=endpoint_geometry["src"],
                    )
                )

            if edge.dst_node_id == node_id:
                local_ports.append(
                    build_incident_port_item(
                        node_id=node_id,
                        edge=edge,
                        end_type="dst",
                        peer_node_id=edge.src_node_id,
                        endpoint=endpoint_geometry["dst"],
                    )
                )

        # 当前正式口径是按 heading_deg_image 排顺时针序。
        # 只是排序单位从 edge 变成了 port，从而允许 same-edge 的 src/dst 被显式区分。
        local_ports.sort(key=heading_deg_image)

        degree = len(local_ports)
        for index, item in enumerate(local_ports):
            item["cw_order_index"] = index
            item["cw_prev_port_id"] = local_ports[(index - 1) % degree]["port_id"] if degree > 0 else None
            item["cw_next_port_id"] = local_ports[(index + 1) % degree]["port_id"] if degree > 0 else None

        port_items.extend(local_ports)

    return {
        "items": tuple(port_items),
        "summary": build_incident_port_summary(port_items),
    }
```

这一步必须写清楚的原因是：

- 第 3 步后面所有 `node_local_connection_hypothesis_info` 都严格依赖这份顺时针 incident port 视图。
- 第 4 步不会再回头从原始 `node.incident_edge_ids` 猜局部进入/离开端语义。
- 如果这里仍然把 same-edge 的 `src / dst` 压成一个 incident edge，第 3 步就无法正确表达 pure cycle cut 和 dead_end。

## 4.5 `build_node_local_connection_hypotheses(...)` 伪代码

```python
def build_node_local_connection_hypotheses(graph_info, incident_port_info, config=None):
    """在单节点内部枚举 in_port -> out_port 连接假设。"""

    # 先解析规则阈值。
    thresholds = resolve_connection_hypothesis_thresholds(config)

    # 只允许从 incident_port_info 里的顺时针 port 视图读取局部事实。
    port_items_by_node = group_incident_ports_by_node(incident_port_info)

    items = []
    items_by_node = {}
    kind_counts = {"through": 0, "turn": 0, "cycle_through": 0, "dead_end_return": 0, "reject": 0}

    for node in graph_info.nodes:
        local_ports = port_items_by_node.get(node.node_id, [])

        local_items = []
        for in_port in local_ports:
            for out_port in local_ports:
                # 同一个 port 自己接自己不构成合法连接假设。
                if in_port["port_id"] == out_port["port_id"]:
                    continue

                connection_kind = classify_node_local_connection_kind(
                    node=node,
                    in_port=in_port,
                    out_port=out_port,
                    thresholds=thresholds,
                )
                if connection_kind is None:
                    continue

                local_items.append(
                    {
                        "hypothesis_id": len(items) + len(local_items) + 1,
                        "via_node_id": node.node_id,
                        "in_port_id": in_port["port_id"],
                        "out_port_id": out_port["port_id"],
                        "in_edge_id": in_port["edge_id"],
                        "in_end_type": in_port["end_type"],
                        "out_edge_id": out_port["edge_id"],
                        "out_end_type": out_port["end_type"],
                        "connection_kind": connection_kind,
                        "base_confidence": resolve_connection_base_confidence(connection_kind),
                        "reason_tags": build_connection_reason_tags(node, in_port, out_port, connection_kind),
                    }
                )

        items.extend(local_items)
        if local_items:
            items_by_node[node.node_id] = local_items
        accumulate_connection_kind_counts(kind_counts, local_items)

    return {
        "items": tuple(items),
        "items_by_node": items_by_node,
        "summary": build_node_local_connection_hypothesis_summary(...),
    }
```

这里需要写细的关键点有三条：

1. node-local hypothesis 的枚举范围严格是“单节点内部 incident port 两两组合”。
2. `connection_kind / base_confidence / reason_tags` 的分类在这一层已经落定，后面不应重判 topology 合法性。
3. 这一步仍然是 topology 层连接假设，不是 sweep 级候选，更不是第 4 步的 final path 语义。

## 4.6 `classify_node_local_connection_kind(...)` 伪代码

```python
def classify_node_local_connection_kind(node, in_port, out_port, thresholds):
    """把 in_port -> out_port 归类为 node-local connection kind。"""

    # 先拦掉不构成真经过边的组合。
    # 这里不再用 `degree < 2` 或 `in_edge == out_edge` 做全局一刀切。
    if not forms_true_traversal(node=node, in_port=in_port, out_port=out_port):
        return None

    # same-edge 的合法性不再一律判非法，而是按节点语义区分。
    if in_port["edge_id"] == out_port["edge_id"]:
        if node.get("is_virtual") and node.get("virtual_reason") == "pure_cycle_cut":
            return "cycle_through"
        if node.get("is_virtual") and node.get("virtual_reason") == "dead_end":
            return "dead_end_return"
        return None

    # 普通 cross-edge 组合继续按局部转向几何分类。
    turn_delta = resolve_port_turn_delta_deg_image(in_port, out_port)
    if abs(turn_delta) <= thresholds["straight_tol_deg"]:
        return "through"
    return "turn"
```

这一层最容易被误解，所以要写死：

- 它是 topology 层正式规则层。
- 它明确替代旧的“same-edge 一律非法”写法。
- 它的价值是把 pure cycle cut / dead_end / 普通路口三种语义统一收进同一套 node-local 连接判定里。

## 4.7 `build_topology_graph_result(...)` 伪代码

```python
def build_topology_graph_result(*, stage_outputs, validation_info):
    graph_info = stage_outputs["graph_info"]
    incident_port_info = stage_outputs["incident_port_info"]
    node_local_connection_hypothesis_info = stage_outputs["node_local_connection_hypothesis_info"]

    return TopologyGraphBuildResult(
        graph_info=graph_info,
        incident_port_info=incident_port_info,
        node_local_connection_hypothesis_info=node_local_connection_hypothesis_info,
        debug_info={
            # 第 3 步当前最核心的 debug 视图是节点局部顺时针 incident port 结果。
            "incident_ports_by_node": group_incident_ports_by_node(incident_port_info),
        },
        validation_info=validation_info,
        meta={
            "stage": "topology_graph_build",
            "node_count": len(graph_info.nodes),
            "edge_count": len(graph_info.edges),
        },
    )
```

这里必须强调两点：

- `node_local_connection_hypothesis_info` 是第 3 步给第 4 步的正式 topology 上游输入。
- 当前第 4 步长期真正应直接读取的第 3 步派生层是它。

## 4.8 第 3 步校验与结果装配伪代码

```python
def build_topology_graph_validation_info(stage_outputs):
    graph_info = stage_outputs["graph_info"]
    incident_port_info = stage_outputs["incident_port_info"]
    node_local_connection_hypothesis_info = stage_outputs["node_local_connection_hypothesis_info"]

    return {
        # 第 3 步第一层先校 graph_info 自己是否闭环。
        "graph_info": validate_graph_info(graph_info),
        # 第二层校 incident port 是否仍然建立在合法 graph 上。
        "incident_ports": validate_incident_ports(
            graph_info=graph_info,
            incident_port_info=incident_port_info,
        ),
        # 第三层校 node-local hypothesis 是否严格遵守真经过边与 connection_kind 规则。
        "node_local_connection_hypotheses": validate_node_local_connection_hypotheses(
            graph_info=graph_info,
            incident_port_info=incident_port_info,
            node_local_connection_hypothesis_info=node_local_connection_hypothesis_info,
        ),
    }
```


## 5. 模块边界

负责：

1. 汇总 `node_info / edge_info`。
2. 建立正式 `graph_info`。
3. 建立节点局部顺时针 `incident_port_info`。
4. 构建 `node_local_connection_hypothesis_info`。
5. 在 topology 层落定 `connection_kind / base_confidence / reason_tags`。
6. 产出第 3 步分层 `validation_info / debug_info / meta`。

不负责：

1. `coverage_lane_info`。
2. `sweeps`。
3. `sweep_transition_candidate_info`。
4. `sweep_cadence_info`。
5. `final_coverage_path_info`。

## 6. 与第 4 步的接口

第 4 步长期真正直接读取的是：

1. `graph_info`
2. `node_local_connection_hypothesis_info`
3. `geometry_preparation_result`
   - 这是通过 `context` 从第 1 步继续传入的，不属于第 3 步输出。

其中：

- `graph_info`
  - 提供 edge 和 node 的正式静态几何与拓扑。
- `node_local_connection_hypothesis_info`
  - 提供后续 sweep transition expansion 所需的 node-local 连接语义。

另外，第 4 步不应继续自己重新判定 pure cycle cut / dead_end 的拓扑合法性；这些语义应由第 3 步先判定并稳定传递下来。

这里要特别区分：

- `incident_port_info`
  - 是第 3 步正式输出的一层，但更偏向节点局部事实视图。
- `node_local_connection_hypothesis_info`
  - 才是第 4 步直接读取的第 3 步 topology 派生层输入。

## 7. 校验重点

### 7.1 已对齐的后续改造约束

在不改写当前源码主链描述的前提下，这里补充三条后续改造约束：

- pure cycle cut、dead_end 的识别与标注，应在第 3 步完成，而不是留到第 4 步 sweep/cadence 再补判。
- 当前源码里“`degree < 2` 直接跳过”以及“`in_edge == out_edge` 直接过滤”的写法，只能视为现状实现，不应继续作为长期合法性规则。
- 长期应从“真经过边”语义判断连接合法性，并显式区分 `cycle_through` 与 `dead_end_return`，而不是把 same-edge 一律当非法或一律并入 `foldback`。

至少检查：

1. `graph_info` 内 `edge/node` 回指闭环。
2. `incident_port_info` 是否与正式节点集合和 edge 端点语义对齐。
3. `node_local_connection_hypothesis_info` 是否仍然严格建立在单节点 incident port 顺时针视图上。
4. same-edge 组合是否只在 `pure_cycle_cut / dead_end` 这些合法语义下保留。
5. `connection_kind` 是否明确区分 `through / turn / cycle_through / dead_end_return`。
6. 不在本阶段提前混入 sweep 级对象。

## 8. 总结

当前第 3 步的正确理解是：

- 先把第 2 步的正式 `node_info / edge_info` 收束成正式 `graph_info`。
- 再补节点局部顺时针 `incident_port_info`。
- 再从 node-local port 组合构建 `node_local_connection_hypothesis_info`。
- 第 4 步后续所有 coverage 逻辑都在 `graph_info + node_local_connection_hypothesis_info` 的基础上继续展开。

这一步的核心不是“提前做 coverage”，而是把 coverage 真正依赖的 topology 语义主线准备干净。
