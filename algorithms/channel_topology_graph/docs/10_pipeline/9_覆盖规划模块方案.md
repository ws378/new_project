# 覆盖规划模块方案

## 1. 文档定位

本文档只描述当前源码中第 4 步 `coverage_planning` 模块的正式主线。

这里有两个硬要求：

1. stage 级对象名必须贴当前源码。
2. stage 级主流程必须贴当前真实装配链。

也就是说，文档可以把职责、字段和边界解释得更细，但不能把当前实现改写成另一套 stage 对象树，也不能把子域内部 helper 误写成 stage 级正式 contract。

本文档直接按已经对齐的 coverage 主线口径组织内容，但保留当前章节结构、伪代码颗粒度和注释密度。

## 3. 当前源码真实主装配链

当前 stage 级真实函数链是：

1. `require_geometry_preparation_result(...)`
2. `build_coverage_lane_sweep_info(...)`
3. `attach_edge_coverage_info(...)`
4. `build_sweep_graph_info(...)`
5. `build_sweep_cadence_info(...)`
6. `build_final_coverage_path_info(...)`
7. `build_coverage_planning_validation_info(...)`
8. `build_coverage_planning_result(...)`

其中第 2 到第 6 步对应四个正式 coverage 子域 build-info。

要特别区分两层东西：

1. stage 级正式主链
   - `build_coverage_lane_sweep_info`
   - `build_sweep_graph_info`
   - `build_sweep_cadence_info`
   - `build_final_coverage_path_info`
2. 子域内部 helper 链
   - `build_coverage_lanes_and_sweeps`
   - `expand_node_local_connection_hypotheses_to_sweep_candidates`
   - `score_sweep_transition_candidates`
   - `build_sweep_cadence`
   - `build_final_coverage_path`

后者当前确实存在，但它们属于子域内部构造链，不应再被误写成 `CoveragePlanning` stage 的正式顶层 contract。

阶段末尾当前还会补三类显式校验：

1. `validate_coverage_lane_generation(...)`
2. `validate_sweep_transition_candidates(...)`
3. `validate_final_coverage_path(...)`

任何设计文档都不应把这条链条改写成源码中不存在的并列主流程。

但也要注意：

- `route_sweep_order`
- `ordered_passes`

已经是当前 `SweepCadenceInfo` 的正式 contract 字段。
它们属于 cadence 真值的展开视图，不是“文档虚构的新世界”。

## 4. 模块边界

### 4.1 负责

- 基于正式 `graph_info` 与 `geometry_preparation_result` 生成 `coverage_lane_sweep_info`
- 把 coverage lane 相关边级真值投影回 `graph_info.edges[*].coverage_info`
- 基于 `coverage_lane_sweep_info + graph_info + node_local_connection_hypothesis_info` 生成 `sweep_transition_candidate_info`
- 基于 `coverage_lane_sweep_info + sweep_transition_candidate_info` 生成 `sweep_cadence_build_info`
- 基于 `graph_info + geometry_result + coverage_lane_sweep_info + sweep_cadence_build_info` 生成 `final_coverage_path_build_info`
- 汇总本阶段的：
  - `debug_info`
  - `validation_info`
  - `meta`

当前这些真值不会再顶层平铺回 `CoveragePlanningResult`，而是分别挂到：

- `coverage_lane_sweep_info`
- `sweep_transition_candidate_info`
- `sweep_cadence_build_info`
- `final_coverage_path_build_info`

### 4.2 不负责

- 不改写第 3 步图的拓扑结构判定。
- 不重建节点和边几何。
- 不重新判定第 3 步已经落定的 topology 合法性真值。
- 不在 stage 入口层直接处理可视化。
- 不把 renderer 逻辑混到算法层。

## 5. 四个正式子域分别回答什么

## 5.1 `build_coverage_lane_sweep_info(...)`

回答：

1. 每条正式 `edge` 如何形成 coverage lane。
2. 每个 coverage lane 上如何离散出正式 `sweeps`。
3. lane + sweep 子域的最小 summary / validation 是什么。

输出真值：

- `coverage_lane_info`
- `sweeps`
- `summary`
- `validation_info`

内部主 helper 当前是：

- `build_coverage_lanes_and_sweeps(...)`

## 5.2 `build_sweep_transition_candidate_info(...)`

回答：

- `node_local_connection_hypothesis_info` 如何展开成 `sweep -> sweep` 候选
- 跨 group 候选如何在 topology 许可下按 sweep 端点几何展开
- 同 group 横移候选如何按有限 rank 邻域展开
- 每个候选的端点距离、sweep 级转角、局部可行性、risk / score 如何计算
- 哪些候选会作为正式候选进入 cadence

输出真值：

- `items`
- `summary`

内部主 helper 当前是：

- `build_sweep_transition_candidates(...)`
- `synthesize_lanes_from_hypotheses(...)`
- `resolve_lane_context(...)`
- `choose_lane_pairs(...)`
- `build_group_internal_candidates(...)`
- `build_candidate_item(...)`

关于连接真值来源，这里要明确：

- 正式主口径直接看 `sweep_transition_candidate_info`
- `connection_kind` 必须继承自第 3 步的 `node_local_connection_hypothesis_info`，不能在第 4 步重新发明
- 第 3 步传下来的 `turn_delta_deg_image` 是 edge / port 级局部转角，只能提供 `motion_type` 期望
- 跨 group pair 是否合理，必须再用 sweep 级 `A/B/C/D` 端点方向和 B/C 距离校验
- 跨 group 以 sweep 端点几何、局部可行性和 top-k 收口为主，并输出多候选集合

## 5.3 `build_sweep_cadence_info(...)`

回答：

- 最终高层 sweep 覆盖节拍骨架是什么
- `route_sweep_order` 是什么
- `ordered_passes` 是什么
- cadence 对 sweep 全集的覆盖率是什么

输出真值：

- `sweep_cadence_info`
- `coverage_stats`
- `summary`

内部主 helper 当前是：

- `build_sweep_cadence(...)`
- `build_sweep_coverage_stats(...)`

## 5.4 `build_final_coverage_path_info(...)`

回答：

- 最终 sweep 段和路口连接段的完整点链是什么
- `ordered_items` 是什么
- `junction_connections` 是什么
- 最终路径的摘要和契约级校验结果是什么

输出真值：

- `final_coverage_path_info`
- `summary`
- `validation_info`

内部主 helper 当前是：

- `build_final_coverage_path(...)`

## 5.5 特殊拓扑语义的接入

针对 pure cycle cut、dead_end 与 same-edge 真经过边，当前文档补充以下约束：

- pure cycle cut / dead_end 的 topology 合法性，不应由第 4 步重新发明，而应由第 3 步先完成识别并传入。
- 第 4 步的 candidate 展开层，后续需要允许 same-edge 的真经过 sweep 连接进入候选，而不是简单按 `same-edge` 过滤。
- 这些 same-edge 候选至少要区分两种语义：`cycle_through` 与 `dead_end_return`。
- cadence 层后续不能再把 “same-edge” 直接等同为非法，也不能直接等同为 `foldback`；只有 `dead_end_return` 才进入受控折返/掉头策略。

这几条就是本阶段正式处理口径，不再是旁路补充说明。

## 6. 当前模块的语义分层

虽然第 4 步对象较多，但当前源码里的语义层级可以明确收成四个 build-info 层：

### 6.1 边级覆盖解释层

- `coverage_lane_sweep_info`

这一层回答：

- 哪些 edge 能转成 coverage lane
- 每条 lane 的局部宽度、有效区、sweep 统计是什么

### 6.2 sweep 静态与粗筛图层

- `sweep_transition_candidate_info`

这一层回答：

- sweep transition 候选如何从 node-local topology 假设展开出来
- 哪些候选会保留进入后续节拍求解
- `connection_kind / selection_level / total_score` 的正式读取口径是什么

### 6.3 节拍层

- `sweep_cadence_build_info`

这一层回答：

- 实际怎么走 sweep 顺序
- 每条 route 的 `segments / start_end_type / end_end_type` 是什么
- cadence 的展开视图 `route_sweep_order / ordered_passes` 是什么

### 6.4 结果层

- `final_coverage_path_build_info`

这一层回答：

- 把 cadence 里的 sweep 段和路口连接段真正展开后，最终点链是什么

## 7. 当前硬约束

### 7.1 geometry 依赖

- `geometry_preparation_result` 必须通过 `context` 传入
- 若缺失，当前 stage 直接 `raise ValueError`

### 7.2 机器人清扫宽度

- `robot_width_m` 当前必须显式给正值
- 若未提供或 `<= 0`，`build_coverage_plan(...)` 直接失败

### 7.3 FinalCoveragePath 的真实输入

当前 `FinalCoveragePath` 直接读取：

- `graph_info`
- `geometry_result`
- `coverage_lane_sweep_info`
- `sweep_cadence_build_info`

不应擅自改写成绕开前面 build-info 层的另一套平铺输入树。

## 8. 伪代码

这一节不再只给一段 stage 级短伪代码，而是按当前正式源码拆成两层：

- 第 1 层是 `CoveragePlanning` stage 主装配主线
- 第 2 层是四个 coverage 子域各自的正式 helper 主线

这样写的目的不是把实现细节全部搬进文档，而是把“正式输入是什么、正式输出是什么、每一层到底负责什么”钉死，避免后续又把 stage 级 contract 和子域内部 helper 混写。

### 8.1 `build_coverage_plan(...)` stage 主装配伪代码

```python
def build_coverage_plan(topology_graph_build_result, config=None, context=None):
    """CoveragePlanning stage 正式入口。"""

    # 入口先把 config / context 规整成普通 dict。
    # 这样下游 helper 不需要兼容 None、Mapping 或外部自定义容器。
    config = dict(config or {})
    context = dict(context or {})

    # CoveragePlanning 的几何真值不是可选输入。
    # geometry_preparation_result 缺失时，当前实现直接视为硬错误。
    geometry_result = require_geometry_preparation_result(context)

    # stage 内部真正的构建主线统一收口到这个 helper。
    # 它负责按固定顺序串行推进四个 coverage 子域。
    stage_outputs = build_coverage_planning_stage_outputs(
        topology_graph_build_result=topology_graph_build_result,
        geometry_result=geometry_result,
        config=config,
    )

    # stage 级 validation 不是重新算一遍业务结果。
    # 它只是把各子域已有 validation / summary 再组织成统一校验视图。
    validation_info = build_coverage_planning_validation_info(stage_outputs)

    # 最终结果对象继续按四个 build-info 子树返回。
    # stage 主装配层不再回退到旧的平铺字段树。
    return build_coverage_planning_result(
        topology_graph_build_result=topology_graph_build_result,
        graph_info=stage_outputs["graph_info"],
        coverage_lane_sweep_info=stage_outputs["coverage_lane_sweep_info"],
        sweep_transition_candidate_info=stage_outputs["sweep_transition_candidate_info"],
        sweep_cadence_build_info=stage_outputs["sweep_cadence_build_info"],
        final_coverage_path_build_info=stage_outputs["final_coverage_path_build_info"],
        validation_info=validation_info,
    )
```

这一层必须明确四件事：

- `CoveragePlanning` 的正式依赖是 `topology_graph_build_result + geometry_preparation_result + config`
- 几何输入从 `context["geometry_preparation_result"]` 读取，不是从别处隐式拿
- stage 主装配只负责串四个 build-info，不直接展开子域内部算法细节
- 最终正式结果是四个 build-info 子树，不是旧文档里的平铺字段集合

### 8.2 `require_geometry_preparation_result(...)` 与 stage 子链路总装配

```python
def require_geometry_preparation_result(context):
    """读取并校验 geometry_preparation_result。"""

    geometry_result = context.get("geometry_preparation_result")

    # 当前 CoveragePlanning 必须读取 free_mask、world 坐标和 resolution。
    # 因此缺失 geometry_result 时不能降级为空，而是直接报错。
    if not isinstance(geometry_result, GeometryPreparationResult):
        raise ValueError("coverage planning requires geometry_preparation_result in context")

    return geometry_result


def build_coverage_planning_stage_outputs(
    *,
    topology_graph_build_result,
    geometry_result,
    config,
):
    """按 coverage lane -> sweep graph -> cadence -> final path 顺序构建。"""

    # 第 4 步第一层：先从 graph edge 真值和 geometry 真值生成 coverage lane / sweep。
    coverage_lane_sweep_info = build_coverage_lane_sweep_info(
        graph_info=topology_graph_build_result.graph_info,
        geometry_result=geometry_result,
        config=config,
    )

    # 第 4 步第二层：把 coverage lane 真值投影回 graph edge 视图。
    # 注意这里不是重建 graph，只是给 edge 附着 coverage 解释信息。
    graph_info = attach_graph_edge_coverage_info(
        topology_graph_build_result=topology_graph_build_result,
        coverage_lane_sweep_info=coverage_lane_sweep_info,
    )

    # 第 4 步第三层到第五层继续严格串行：
    # sweep transition candidate 先于 sweep cadence，sweep cadence 又先于 final coverage path。
    sweep_transition_candidate_info, sweep_cadence_build_info, final_coverage_path_build_info = (
        build_downstream_coverage_outputs(
            graph_info=graph_info,
            geometry_result=geometry_result,
            coverage_lane_sweep_info=coverage_lane_sweep_info,
            topology_graph_build_result=topology_graph_build_result,
            config=config,
        )
    )

    return {
        "graph_info": graph_info,
        "coverage_lane_sweep_info": coverage_lane_sweep_info,
        "sweep_transition_candidate_info": sweep_transition_candidate_info,
        "sweep_cadence_build_info": sweep_cadence_build_info,
        "final_coverage_path_build_info": final_coverage_path_build_info,
    }
```

这里要写死的约束是：

- 三个下游 coverage 子域必须按顺序串行推进，不能随意交叉互读临时变量
- `graph_info` 在这一阶段只做“附着 coverage edge 解释信息”，不重新生成拓扑图主干
- `final_coverage_path_build_info` 是 CoveragePlanning 内部最后一个正式 build-info 层

### 8.3 `build_coverage_lane_sweep_info(...)` 伪代码

```python
def build_coverage_lane_sweep_info(graph_info, geometry_result, config=None):
    """从 graph edge 和 geometry 真值构建 coverage lane / sweep。"""

    config = dict(config or {})

    # 先从 geometry_result 里读取第 4 步必需的几何真值。
    # 这里正式依赖 free_mask / obstacle_mask / resolution 等事实。
    free_mask = geometry_result.free_mask
    obstacle_mask = geometry_result.obstacle_mask
    resolution_m_per_px = geometry_result.resolution_m_per_px

    # coverage lane sweep 的内部主 helper 负责真正做几何采样、宽度估计和 sweep 生成。
    # 这一层返回的是 coverage lane 域自己的完整真值，而不是 graph 的一部分。
    lane_generation_result = build_coverage_lanes_and_sweeps(
        graph_info=graph_info,
        free_mask=free_mask,
        obstacle_mask=obstacle_mask,
        resolution_m_per_px=resolution_m_per_px,
        config=config,
    )

    coverage_lane_info = lane_generation_result["coverage_lane_info"]
    sweeps = lane_generation_result["sweeps"]

    # validation 在这一层主要回答“哪些 edge 成功生成 coverage lane，哪些失败，失败原因是什么”。
    validation_info = validate_coverage_lane_generation(
        graph_info=graph_info,
        coverage_lane_info=coverage_lane_info,
        sweeps=sweeps,
    )

    return {
        "coverage_lane_info": coverage_lane_info,
        "sweeps": sweeps,
        "summary": "<按 coverage_lane_sweep_info 当前规则组装的摘要>",
        "validation_info": validation_info,
    }
```

这一层必须强调：

- 这里回答的是“edge 如何转成 coverage lane / sweep”，不是 sweep 之间怎么连接
- `coverage_lane_info` 和 `sweeps` 是第 4 步最底层的边级覆盖真值
- `validation_info` 在这里聚焦 coverage lane 生成质量，不冒充 sweep graph 或 final path 校验

### 8.4 `attach_graph_edge_coverage_info(...)` 伪代码

```python
def attach_graph_edge_coverage_info(topology_graph_build_result, coverage_lane_sweep_info):
    """把 coverage lane 真值挂回 graph edge 视图。"""

    # 这里只做 edge coverage 投影，不改 node/edge 主身份字段。
    # graph 的拓扑结构仍然继承自 topology_graph_build_result.graph_info。
    return attach_edge_coverage_info(
        graph_info=topology_graph_build_result.graph_info,
        coverage_lane_info=coverage_lane_sweep_info.coverage_lane_info,
    )
```

这一层的边界要写清楚：

- 它不是新的拓扑构建步骤
- 它不重新生成 `node_info_list / edge_info_list`
- 它的职责只是让后续 coverage 子域能从 `graph_info.edges[*]` 直接读取 coverage 附着信息

### 8.5 `build_downstream_coverage_outputs(...)` 伪代码

```python
def build_downstream_coverage_outputs(
    *,
    graph_info,
    geometry_result,
    coverage_lane_sweep_info,
    topology_graph_build_result,
    config,
):
    """串行构建 sweep transition candidate、sweep cadence、final coverage path。"""

    # sweep transition candidate 是后面两个子域的前置层。
    sweep_transition_candidate_info = build_sweep_transition_candidate_info(
        graph_info=graph_info,
        node_local_connection_hypothesis_info=topology_graph_build_result.node_local_connection_hypothesis_info or {},
        coverage_lane_sweep_info=coverage_lane_sweep_info,
    )

    # cadence 只消费 coverage lane sweep 与 sweep transition candidate 两层正式真值。
    # cadence 配置通过独立 config helper 收口，避免直接耦合外部大配置树。
    sweep_cadence_build_info = build_sweep_cadence_info(
        coverage_lane_sweep_info=coverage_lane_sweep_info,
        sweep_transition_candidate_info=sweep_transition_candidate_info,
        config=build_sweep_cadence_config(config),
    )

    # final coverage path 是 CoveragePlanning 内部最后一个正式构建层。
    # 它要同时读取 graph、geometry、coverage lane、cadence 四类输入真值。
    final_coverage_path_build_info = build_final_coverage_path_info(
        graph_info=graph_info,
        geometry_result=geometry_result,
        coverage_lane_sweep_info=coverage_lane_sweep_info,
        sweep_cadence_build_info=sweep_cadence_build_info,
        config=build_final_coverage_path_config(config, geometry_result),
    )

    return (
        sweep_transition_candidate_info,
        sweep_cadence_build_info,
        final_coverage_path_build_info,
    )
```

这一段要钉住三件事：

- `sweep_transition_candidate_info -> sweep_cadence_build_info -> final_coverage_path_build_info` 是严格前后依赖关系
- `node_local_connection_hypothesis_info` 是第 3 步给第 4 步的正式 topology 上游输入之一
- `final_coverage_path_build_info` 不能绕开前面 build-info 子树，另起一套隐式输入链

### 8.6 `build_sweep_transition_candidate_info(...)` 伪代码

```python
def build_sweep_transition_candidate_info(graph_info, node_local_connection_hypothesis_info, coverage_lane_sweep_info):
    """把 topology node-local 假设展开成 sweep transition candidate 真值。"""

    sweeps = coverage_lane_sweep_info.sweeps

    # 第一步先从 sweeps 建立 group、port view 等静态索引。
    # group/port 提供 topology 许可和端点视图，跨 group 几何对应由 sweep 端点几何裁决。
    sweep_group_info = build_sweep_group_info(graph_info=graph_info, sweeps=sweeps)
    sweep_port_view_info = build_sweep_port_view_info(sweep_group_info=sweep_group_info)

    # 第二步把 topology node-local hypothesis 作为跨 group 连接意图输入。
    # topology 层的 turn_delta_deg_image 只说明 edge/port 级 motion_type 期望。
    # 具体 sweep pair 是否合理，必须在 candidate 构造中用 sweep 级端点几何再判断。

    # 第三步生成正式 sweep transition candidates。
    # 跨 group:
    #   - 在 topology 许可的 group-pair 内展开多候选。
    #   - 用 A/B/C/D 计算 B/C 端点距离与 sweep 级转角。
    #   - 以 sweep 端点几何和局部可行性为主。
    #   - 输出可供 cadence 选择的多候选集合。
    #   - 当前没有 rank frame 对齐，不能用 rank_gap / mean_offset_m / side_level 主导选择。
    # 同 group:
    #   - 可以使用同一 port view 内的 rank_gap 做有限横移邻域。
    #   - rank_gap / mean_offset_m / side_level 处在同一 group frame 内，但仍要结合端点几何。
    candidate_info = build_sweep_transition_candidates(
        sweep_group_info=sweep_group_info,
        sweep_port_view_info=sweep_port_view_info,
        sweeps=sweeps,
        node_local_connection_hypothesis_info=node_local_connection_hypothesis_info,
    )

    return {
        "items": tuple(candidate_info["items"]),
        "summary": candidate_info["summary"],
    }
```

这一层需要特别写明：

- `build_sweep_transition_candidate_info(...)` 回答的是“topology 假设如何展开成 sweep 候选”，不是“最终清扫顺序怎么走”
- `items` 是 cadence 的唯一正式候选读取口径
- 关于连接真值来源，当前正式口径直接读 `sweep_transition_candidate_info`，并继承第 3 步传下来的 `connection_kind`
- 跨 group 候选阶段保留基于 sweep 端点几何和局部可行性的多候选选择空间
- cadence 对跨 group candidate 做最终选择时，pair 连接质量必须先于目标 sweep 角色偏好

## 9. 总结

当前 `CoveragePlanning` 模块方案必须同时守住三件事：

1. stage 级正式主链是四个 build-info，不是旧平铺对象树。
2. 子域内部 helper 可以继续存在，但不能冒充 stage 级正式 contract。
3. `route_sweep_order / ordered_passes / sweep_transition_candidate_info` 已经进入正式读取口径，文档不能继续按旧的中间连接壳对象处理。
4. pure cycle cut / dead_end 的新语义必须通过 `node_local_connection_hypothesis_info -> sweep_transition_candidate_info -> cadence` 这条主线传递下去。
