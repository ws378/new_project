# 代码事实

`02_代码事实` 只保存能长期约束代码维护的事实边界。它回答“当前代码和证据资产怎样被使用、怎样禁止误用”，不回答“某次扫描得到了什么结果”，也不逐项复述目录里每个脚本。

## 1. 职责

本目录只承担三类职责：

- 明确 formal planner、UI 入口、研究包和第三方复现代码之间的依赖边界；
- 明确研究脚本、诊断脚本、归档脚本的生命周期，防止研究入口误接入正式链路；
- 明确 output 证据资产的索引、保留、归档和删除规则。

## 2. 文件组织

| 文件 | 长期职责 | 不负责 |
| --- | --- | --- |
| `01_正式代码入口与依赖边界.md` | 说明正式代码链路、研究包边界、允许依赖和禁止依赖。 | 不列全量脚本清单，不记录批次状态。 |
| `02_研究代码与脚本生命周期.md` | 说明 baseline、diagnostics、experiments、archived 的语义和迁移规则。 | 不保证每个历史脚本长期可执行。 |
| `03_output证据资产规则.md` | 说明 output 证据资产如何索引、引用、归档和删除。 | 不保存单次 manifest 扫描摘要。 |

## 3. 禁止内容

- 不保存自动扫描生成的时间戳、数量统计和一次性候选表；
- 不把执行记录、历史看板或旧批次状态当作当前代码事实；
- 不把研究脚本列表写成 formal planner 能力；
- 不用“当前目录里有什么文件”替代“这些文件在工程中承担什么责任”；
- 不保存已经被长期规则吸收的删除复核长表。

## 4. 与其他目录的关系

- 算法思路、数据结构和主流程放在 `01_主题/`；
- 当前缺口和拒绝路线放在 `03_缺口/`；
- 推进约束、删除吸收核查和治理规则放在 `04_开发/`；
- 测试分层、验收清单和当前基线放在 `05_验证/`；
- 论文、第三方来源和人工标注放在 `reference/`。

如果某个文档只是说明“我这次做了什么”，它不属于 `02_代码事实`。
