# 机器人几何与覆盖定义输入契约

## 1. 目标

本文定义 `shelfAware` 和 `ShelfAware+TurnCost` 做只读几何覆盖诊断时必须具备的输入契约，包括路径参考点、车体硬结构、清扫部件、目标区域和路径姿态。

本文只定义输入和语义，不修改路径规划算法。

## 2. 坐标约定

建议统一使用机器人局部坐标系：

- 原点：后轴中心 `O`；
- `+x`：机器人右侧；
- `+y`：机器人前进方向；
- 路径点：默认表示后轴中心轨迹；
- 路径 `yaw`：默认表示机器人车体朝向。

如果工程实际使用 `base_link`、几何中心、雷达中心或其他参考点，必须在配置中显式声明，并转换到本文坐标系后再做诊断。

## 3. 路径输入

每个诊断 case 至少需要：

```json
{
  "path": {
    "reference_point": "rear_axle_center",
    "pose_source": "path_pixels.json",
    "coordinate_frame": "image_pixel",
    "yaw_source": "from_planner_or_estimated_from_path",
    "poses": [
      {
        "index": 1,
        "x": "<pixel_x>",
        "y": "<pixel_y>",
        "theta": "<rad>"
      }
    ]
  }
}
```

字段说明：

- `reference_point`：路径点对应的机器人参考点。当前契约推荐 `rear_axle_center`。
- `pose_source`：路径数据来源，通常是 UI 或实验输出的 `path_pixels.json`。
- `coordinate_frame`：路径坐标系。当前 MapTools 诊断优先使用 `image_pixel`。
- `yaw_source`：姿态来源。若 planner 未提供可靠 yaw，应按相邻路径点估算，并记录为 `estimated_from_path`。
- `poses[].theta`：机器人车体朝向，单位弧度。图像坐标系下的正负号必须与路径生成端保持一致。

路径诊断必须保留原始 `index`，保证风险窗口能回指到路径点。

## 4. 地图与目标区域输入

每个诊断 case 至少需要：

```json
{
  "map": {
    "resolution_m": "<地图分辨率>",
    "origin": ["<origin_x>", "<origin_y>"],
    "free_mask": "<02_free_mask.png>",
    "prepared_map": "<05_prepared_map.png>",
    "region_mask": "<region_mask.png>",
    "collision_mask_policy": "free_mask_or_prepared_map",
    "target_mask_policy": "region_mask AND prepared_map"
  }
}
```

字段说明：

- `resolution_m`：地图分辨率，单位 m/px。
- `origin`：地图原点，用于需要世界坐标输出时转换。
- `free_mask`：可通行区域原始或基础 mask。
- `prepared_map`：经过既有预处理后的规划可用区域。
- `region_mask`：当前区域多边形 raster 化结果。
- `collision_mask_policy`：车体碰撞诊断使用的障碍口径。当前必须在输出中写明使用 `free_mask` 还是 `prepared_map`。
- `target_mask_policy`：清扫覆盖目标区域口径。推荐使用 `region_mask AND prepared_map`，避免把障碍内部计入漏扫。

## 5. 碰撞 footprint 输入

碰撞 footprint 只包含硬结构，用于判断机器人能不能走。

最小输入：

```json
{
  "body": {
    "reference_point": "rear_axle_center",
    "rear_axle_to_front_m": "<标定值>",
    "rear_axle_to_rear_m": "<标定值>",
    "rear_axle_to_left_m": "<标定值>",
    "rear_axle_to_right_m": "<标定值>"
  }
}
```

字段说明：

- `rear_axle_to_front_m`：后轴中心到车体前向边界的距离。
- `rear_axle_to_rear_m`：后轴中心到车体后向边界的距离。
- `rear_axle_to_left_m`：后轴中心到车体左侧边界的距离。
- `rear_axle_to_right_m`：后轴中心到车体右侧边界的距离。

诊断时应检查车体矩形沿路径和姿态变化后的 swept footprint，而不是只检查路径中心线。碰撞口径优先复用既有预处理和障碍膨胀结果，不新增与规划参数重复的外扩语义。

若需要输出紧贴障碍风险，可在诊断配置中设置 `clearance_warning_m`。它只用于报告“接近障碍”的提示阈值，不参与路径规划，不替代障碍膨胀参数。

## 6. 清扫 footprint 输入

清扫 footprint 用于判断覆盖质量，不等同于碰撞 footprint。

清扫 footprint 至少包含：

- 左盘刷圆；
- 右盘刷圆；
- 刮水器软覆盖弧线或多段线。

建议输入：

```json
{
  "cleaning": {
    "brushes": [
      {
        "name": "left_disc_brush",
        "center_x_m": "<标定值>",
        "center_y_m": "<标定值>",
        "radius_m": "<标定值>"
      },
      {
        "name": "right_disc_brush",
        "center_x_m": "<标定值>",
        "center_y_m": "<标定值>",
        "radius_m": "<标定值>"
      }
    ],
    "squeegee": {
      "enabled_for_coverage": true,
      "enabled_for_collision": false,
      "type": "arc_or_polyline",
      "points_m": [],
      "can_extend_beyond_body_width": true,
      "effective_width_m": "<标定值>"
    }
  }
}
```

字段说明：

- `brushes[].center_x_m` / `center_y_m`：盘刷圆心相对后轴中心的位置。
- `brushes[].radius_m`：盘刷有效覆盖半径。
- `squeegee.enabled_for_coverage`：刮水器是否参与覆盖率计算。
- `squeegee.enabled_for_collision`：刮水器是否参与硬碰撞检查；当前契约约定为 `false`。
- `squeegee.type`：刮水器几何表达方式，当前支持 `arc_or_polyline`。
- `squeegee.points_m`：刮水器弧线或多段线采样点，相对后轴中心。
- `squeegee.can_extend_beyond_body_width`：刮水器软覆盖边是否允许横向超出刚性车体宽度；当前契约按软覆盖部件处理，应允许。
- `squeegee.effective_width_m`：刮水器有效覆盖宽度。

## 7. 现有覆盖定义

现有算法和质量守卫中常用的是路径中心线 buffer 覆盖：

```text
coverage_area ~= buffer(path_centerline, coverage_width_m / 2)
```

这个定义简单、稳定、便于快速比较，但它不是严格真实覆盖。

相关参数语义必须区分：

- `robot_width_m`：机器人通行宽度，属于既有预处理、障碍膨胀和路径可通行性口径。
- `coverage_width_m`：中心线 buffer 覆盖宽度，只用于旧覆盖率指标和对照诊断。
- 真实清扫覆盖：由盘刷和刮水器 swept footprint 计算，不由 `coverage_width_m` 直接替代。

它的局限：

- 默认路径点接近清扫中心线，但后轴中心参考点路径不满足该假设；
- 默认清扫宽度在路径左右对称，但盘刷和刮水器可能不完全对称；
- 对转弯、斜插、路口和端部覆盖的估计可能偏差较大；
- 无法区分盘刷贡献和刮水器贡献。

## 8. 推荐覆盖定义

更严谨的覆盖应定义为：

```text
cleaning_coverage_area =
  swept_area(left_brush_circle, path_pose_sequence)
  ∪ swept_area(right_brush_circle, path_pose_sequence)
  ∪ swept_area(squeegee_soft_footprint, path_pose_sequence)
```

然后：

```text
cleaning_footprint_coverage_ratio =
  area(cleaning_coverage_area ∩ target_area) / area(target_area)
```

推荐 `target_area = region_mask AND prepared_map`。

该定义能回答：

- 盘刷实际扫到了哪里；
- 刮水器实际覆盖了哪里；
- `coverage_width_m` buffer 覆盖率与真实清扫覆盖率差多少；
- 某些线距不均是否真的造成漏扫；
- 某些看似过密区域是否只是视觉重复，还是清扫 footprint 真正重复过多。

## 9. 诊断输出边界

当前只读诊断至少输出：

- 指标 summary；
- 车体平移 swept footprint 风险图；
- 转弯/掉头 swept footprint 风险图；
- 清扫 footprint 覆盖与漏扫图；
- 中心线 buffer 与真实清扫覆盖差异图；
- 风险窗口 JSON。

本文只规定诊断输出必须能回指输入来源和关键口径，不维护输出字段 schema。输出字段真值由 `01_主题/03_机器人几何与覆盖定义/03_两种正式方案的几何覆盖诊断方案.md` 维护；转角窗口字段计算示例由 `01_主题/03_机器人几何与覆盖定义/04_转角窗口字段计算说明.md` 维护；当前实现和样例证据由 `05_验证/05_只读几何诊断验证记录.md` 维护。

诊断输出必须明确：

- 使用的路径来源；
- 使用的机器人几何参数来源；
- 使用的碰撞 mask 口径；
- 使用的目标清扫区域口径；
- yaw 是 planner 原始输出还是估算值；
- 是否包含转弯/掉头 yaw 插值扫掠。

风险窗口 JSON 必须能回指真实路径点，不只输出全局计数。字段含义、合并规则和主指标归属不在本文重复定义。

## 10. 对两种正式方案的约束

### 10.1 `shelfAware`

`shelfAware` 仍可作为稳定 UI 入口，但应补充同一套只读诊断：

- 车体平移 swept footprint 碰撞；
- 车体转弯/掉头 swept footprint 碰撞；
- 车体紧贴障碍；
- 盘刷覆盖率；
- 刮水器覆盖率；
- buffer 覆盖率和真实清扫覆盖率差异。

诊断失败不应立即改规划结果，但应作为是否进入下游执行链路的风险证据。

### 10.2 `ShelfAware+TurnCost`

`ShelfAware+TurnCost` 路径结构更规整，但仍需要同一套诊断：

- 不能因为 turn cost 更低，就认为车体扫掠安全；
- 不能因为 lane spacing 更均匀，就认为真实清扫 footprint 覆盖均匀；
- CTG auxiliary 不可用时仍会生成路径，诊断中要保留该状态。

## 11. 当前实现边界

当前契约只覆盖只读诊断，不直接修改规划：

- 输入：已有路径、地图、区域、机器人几何配置。
- 输出：指标表、诊断图、风险窗口 JSON。
- 不把诊断结果回写路径。
- 不把刮水器纳入硬碰撞。
- 不基于单个地图或单个区域做专用补丁。

## 12. 发布级使用前必须确认的参数

发布级使用几何诊断结论前，必须确认以下真实机器人参数：

- 后轴中心到车体前边界距离；
- 后轴中心到车体后边界距离；
- 车体左右半宽；
- 障碍膨胀结果和紧贴风险诊断阈值；
- 盘刷圆心位置和半径；
- 刮水器形状、有效宽度和是否参与覆盖；
- 路径点在控制器中到底表示后轴中心、`base_link` 还是其他参考点；
- yaw 是否严格表示车体朝向；
- 控制器是否会对折线做平滑、限曲率或原地旋转。
