# 覆盖图与节点生成模型

## 1. 设计目标

覆盖图是 `shelf_aware_turn_cost` 的核心模型。它要把“在哪里覆盖、按什么方向覆盖、相邻覆盖 lane 如何保持稳定、哪些连接是正常遍历”前移到路径生成阶段，而不是等最终路径出来后再批量修补。

长期目标：

- 用规则覆盖单元表达覆盖候选；
- 让 lane 间距尽量接近覆盖宽度；
- 在走廊和货架行方向上保持路径方向稳定；
- 为遍历评分提供转角、方向、间距和来源信息；
- 为最终路径 provenance 提供可追溯数据。

## 2. 当前 `CoverageCell` / `CellCandidate` 字段

`CoverageCell` 表示一个覆盖候选单元。它是静态结构，不记录遍历动态状态。当前代码事实以 `coverage_graph.py` 中的 `CoverageCell` 和构图前的 `CoverageCellLike` / cell candidate 字段为准。

当前已经具备的长期可依赖字段：

| 字段 | 语义 |
| --- | --- |
| `cell_id` | 覆盖单元稳定身份。 |
| `grid_row` / `grid_col` | 规则网格行列。 |
| `grid_center_px` | 原始规则网格中心点。 |
| `planning_point_px` | 经过生成策略和受限修复后的规划参考点。 |
| `obstacle` | 该单元是否不可覆盖或不可访问。 |
| `obstacle_ratio` / `obstacle_ratio_filtered` | 节点生成阶段的障碍占比和过滤状态。 |
| `adjusted_from_grid_center_px` | 是否从规则中心偏移。 |
| `generated_planning_point_px` | 节点生成阶段产出的参考点。 |
| `generation_offset_from_grid_center_px` / `generation_offset_distance_px` | 从规则中心到生成点的偏移。 |
| `generation_mode` / `generation_status` | 节点生成模式和生成状态。 |
| `endpoint_alignment_applied` / `endpoint_alignment_offset_px` | 起止航段对齐是否应用及偏移。 |
| `neighbor_cell_ids` / `accessible_neighbor_cell_ids` | 静态邻接关系和可访问邻接关系。 |

禁止把 `visited`、`visit_count`、当前候选评分、父节点等遍历状态写入 `CoverageCell`。

## 3. 长期目标字段

以下字段是后续模型方向，不得在当前实现中假定已经存在。若要迁入主流程，必须先补代码结构、diagnostics、基线和验收门禁。

| 字段 | 目标语义 | 当前状态 |
| --- | --- | --- |
| `axis_angle` | 候选覆盖方向或局部主方向。 | 当前由方向场和评分组件间接使用，尚未成为 cell 静态字段。 |
| `lane_id` / `lane_family_id` | 所属 lane 或局部 lane family。 | 属于 `lane family / topology source` 缺口。 |
| `coverage_width_m` | 该单元使用的覆盖宽度语义。 | 当前由 planner/profile 参数持有，尚未逐 cell 固化。 |
| `coverage_contribution` | 该单元或路径段的独立覆盖贡献。 | 属于 coverage contribution 缺口。 |
| `quality_hint` | 贴边、窄通道、低置信方向等静态质量提示。 | 当前部分由 diagnostics 推断，尚未统一进入图模型。 |

## 4. `CoverageGraph`

`CoverageGraph` 是覆盖单元和候选相邻关系的静态图。

它应负责：

- 查询 cell；
- 查询邻接候选；
- 查询 lane / family 信息；
- 提供方向、距离、边界和可达性静态信息；
- 为遍历阶段提供候选 move 的静态基础。

它不应负责：

- 记录访问状态；
- 决定遍历阶段一定选择哪个候选；
- 在查询时修改旧 `Node`；
- 根据最终路径质量临时改变静态图。

## 5. 节点生成策略

当前正式候选增强路线使用 `turn_cost_repaired_grid`。该策略的价值不是“复刻论文官方节点”，而是把规则覆盖网格和局部修复结合起来，减少 shelf-aware 纯局部节点导致的线距不稳。

生成过程应包含：

1. 从已有预处理结果得到可覆盖区域；
2. 根据 coverage width 和局部方向生成规则候选；
3. 对边界、窄通道和局部缺口做受限修复；
4. 给每个候选点记录来源和偏移量；
5. 构造 lane / family 关系；
6. 生成覆盖图邻接关系。

受限修复必须有边界：

- 偏移不能超过 profile 允许范围；
- 修复不能制造障碍穿越；
- 修复不能因为局部线距好看而牺牲长跳跃、覆盖率或起止对齐；
- 修复结果必须可以被 diagnostics 解释。

## 6. 线间距稳定性

用户指出的核心问题是“线和线之间横向距离不稳定”，不是路径点间距不一致。因此线距指标应围绕 lane 横向关系，而不是相邻路径点距离。

需要关注：

- 过密 lane 数；
- 过疏 lane 数；
- 左右不平衡窗口；
- 局部 lane family 的稳定性；
- 路口和 junction 区域的可容忍扰动；
- 删除或合并 lane 后是否造成漏扫和长跳跃。

线距均衡的优先顺序：

1. 节点生成阶段让 lane family 更稳定；
2. 遍历评分阶段偏好稳定 lane 连接；
3. 只在受限路径实现阶段处理可证明安全的小范围几何调整；
4. 不做全路径点吸附和无约束 lane 删除。

## 7. 方向场与 CTG 辅助

方向场、货架行方向、局部障碍结构和 CTG territory / junction 都是辅助信息，不是单独主流程。

使用原则：

- 有高置信方向时，用于节点生成和候选方向偏置；
- 置信度低或坐标系不一致时，显式降级；
- CTG topology 报短边或 smoothing 失败时，可以有记录地关闭对应辅助层，但不能改变主算法身份；
- 所有辅助输入都要进入参数快照和 diagnostics。

## 8. 旧 `Node` 边界

旧 `Node` 可以作为 legacy shell 或 semantic 过渡边界存在，但不能继续作为长期模型主体，也不能承担动态访问状态。允许保留的前提是：

- 它只作为旧接口适配、静态语义查询或外部依赖的壳；
- 动态遍历状态只由 `TraversalState` 或等价结构管理；
- 不写入、不同步、不同源维护 `Node.visited` / `Node.visit_count`；
- 不新增依赖旧 `Node` 动态字段的算法逻辑。

如果后续继续重构，应优先减少旧 `Node` 对静态语义和外部接口的残余表达，而不是恢复动态 mirror。
