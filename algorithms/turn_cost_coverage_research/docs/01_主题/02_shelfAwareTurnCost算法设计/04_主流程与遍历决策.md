# 主流程与遍历决策

## 1. 阶段流水线

`shelf_aware_turn_cost` 的主流程应保持线性、可追溯、可失败解释。推荐长期阶段如下：

1. `InputValidation`：校验输入、参数、坐标系和已有预处理结果；
2. `WorkspacePreparation`：建立必要的旋转坐标、裁剪区域和尺度信息；
3. `GuidanceField`：读取方向场、货架行方向和 CTG 辅助信息；
4. `CoverageGraphBuild`：生成覆盖单元、lane family 和邻接图；
5. `StartSelection`：选择起点附近可解释的初始覆盖单元；
6. `TraversalPlanning`：基于候选评分和转角代价生成遍历序列；
7. `FinalPathRealization`：把遍历序列实现成正式路径点；
8. `OutputAssembly`：生成 `CoveragePlanningResult`、diagnostics 和 artifact manifest。

每个阶段输出一个结构化结果。阶段结果可以被 diagnostics 引用，但不能在多个地方形成双真值。

## 2. 状态机边界

遍历动态状态应由 `TraversalState` 或等价结构持有。它至少表达：

- 已访问 cell；
- 访问次数；
- 当前 cell；
- 当前 heading；
- 最近 move 来源；
- 当前 lane / family；
- 必要的 debug 快照。

静态图不写动态状态。旧 `Node` 若仍存在，只能作为 legacy shell 或外部接口适配；不得承担动态状态 mirror，也不得被多个阶段写入访问状态。

## 3. 候选 move

遍历阶段不应直接在裸坐标点之间跳转，而应基于候选 move 决策。

候选 move 至少需要包含：

- `from_cell_id`；
- `to_cell_id`；
- 几何距离；
- heading 变化；
- lane / family 关系；
- 是否穿越 junction 或边界区域；
- 来源标签；
- 可行性状态；
- 评分分解。

来源标签只能解释路径来源，不能演化成独立主算法分支。

## 4. 评分分解

候选评分需要能解释“为什么选这一步”。当前代码事实以 `candidate_scoring.py` 的 `SCORE_COMPONENT_*` 为准；长期目标语义只能通过映射表达，不能把尚未实现的分量写成当前 schema。

当前评分组件族：

| 分量 | 当前语义 |
| --- | --- |
| `distance_cost` | 基础移动距离。 |
| `turn_cost` | 当前方向到候选方向的转角代价。 |
| `local_direction_cost` | 与局部方向场的方向偏差。 |
| `legacy_horizontal_bias` | 旧水平偏置项，仅作为当前代码事实保留，不应继续扩展。 |
| `local_lateral_cost` | 与局部方向相关的横向偏置。 |
| `ctg_same_edge_reward` / `ctg_edge_switch_cost` / `ctg_fallback_edge_switch_cost` / `ctg_junction_entry_cost` | CTG 辅助下的同边、换边、fallback 换边和 junction 入口偏置。 |
| `fallback_jump_cost` / `fallback_heading_cost` | fallback 候选的距离和方向惩罚。 |
| `local_residual_continue_reward` / `local_residual_leave_cost` | 局部 residual 区域继续或离开的偏置。 |
| `revisit_penalty_cost` / `revisit_frontier_reward` | 重复访问惩罚和 frontier 奖励。 |
| `history_clearance_cost` | 与历史轨迹距离相关的避让代价。 |

目标评分语义映射：

| 目标语义 | 当前代码组件或状态 | 长期缺口 |
| --- | --- | --- |
| lane 连续性 | `local_direction_cost`、`local_lateral_cost`、部分 CTG 组件。 | 缺少显式 `lane_id / lane_family_id`。 |
| 线间距风险 | `history_clearance_cost` 和只读 lane spacing diagnostics 可间接暴露。 | 缺少正式 `spacing_risk_cost` 和局部 lane family 结构。 |
| 新覆盖贡献 | 访问状态和 frontier 奖励只能近似表达。 | 缺少正式 `coverage_gain / coverage_contribution`。 |
| 贴边、窄通道和异常转移风险 | fallback、CTG、diagnostics 可部分表达。 | 缺少统一 `risk_penalty` schema。 |

当前候选 rejection 只显式区分：

| reason | 当前语义 | 边界 |
| --- | --- | --- |
| `shape_constraint` | 候选段违反形态约束。 | 只说明该候选不可接受，不等同于全局几何安全证明。 |
| `turn_constraint` | 候选段违反转角约束。 | 与 fallback relaxed turn constraint 配合使用，不能替代完整运动学约束。 |

fallback 的转角约束存在分层放宽：近邻候选使用更严格角度，距离达到 `fallback_relax_dist_m` 后允许放宽到 `fallback_max_turn_deg`。这属于当前代码事实，用于避免局部无候选时完全卡死；它不是新的主算法分支，也不能被扩展成无约束补丁集合。

评分项必须有明确量纲或归一化方式，避免某个经验权重压倒全部目标。

## 5. 转角代价

转角代价是本方案区别于普通 shelf-aware 路径连接的重要内容。它应当影响候选选择，而不是只在最终路径图上做统计。

转角代价需要同时关注：

- 单点急转；
- 短距离连续折线；
- 7 点窗口级方向突变；
- lane endpoint 的掉头风险；
- junction 区域允许的合理转向；
- 长跳跃和 fallback 连接后的方向突变。

这些指标的诊断细节放在机器人几何和验证文档中；主流程只负责记录足够的候选评分和路径来源。

## 6. connector / fallback 的语义

connector、fallback、postprocess 不是三条新主线。它们在本设计里只能表达：

- move 或 path segment 的来源；
- 候选评分中的风险项；
- diagnostics 中的风险归因；
- 极少数受限实现阶段的必要转换。

其中：

- `coverage_like`：正常覆盖推进；
- `connector_like`：连接两个覆盖结构的路径段；
- `fallback_like`：正常候选不足时的低优先级合法转移；
- `final_transform`：最终路径实现阶段记录的受限转换。

如果某个新功能需要大量依赖 connector/fallback 才能跑通，说明覆盖图或遍历决策设计不自然，应回到主流程建模，而不是继续扩展补丁集合。

## 7. 失败与降级

允许降级，但必须显式记录：

- 哪个阶段失败；
- 失败原因；
- 是否影响路径输出；
- 是否切换辅助层；
- 是否改变 profile 行为；
- 是否需要进入验收风险。

不允许静默降级为另一套 planner，也不允许把失败后生成的路径标成同一 profile 的正常结果。

## 8. 效果守卫

遍历决策和路径实现变更至少不能恶化：

- 覆盖率；
- 窄通道覆盖；
- 长跳跃数量；
- 不可行段数量；
- lane 过密、过疏和左右不平衡；
- 总转角和急转热点；
- 几何只读风险；
- 运行时间。

守卫阈值和固定用例在 `docs/05_验证/` 维护，不在本主题文档中重复写死。
