# 算法定位与边界

## 1. 核心定位

`shelf_aware_turn_cost` 是当前 UI 可选的候选增强覆盖规划方案。它保留 `shelf_aware` 对地图预处理结果、货架/走廊局部方向和工程参数的使用方式，同时把 turn-cost 研究中对路径形态有确定价值的部分前移到主流程。

它解决的主要问题是：

- coverage lane 横向间距不稳定；
- 局部路径交叉、短折返、斜插过多；
- 转角和急转热点偏多；
- 路径来源不透明，后续质量诊断难以对应真实生成阶段；
- 后处理实验容易局部改善但破坏长跳跃、覆盖率或线距均衡。

因此本方案的核心原则是：优先让覆盖图和遍历决策自然产生更稳定的路径，而不是在最终路径上叠加批量补丁。

## 2. 与其他模式的关系

| mode | 定位 | 是否是本目录主题 |
| --- | --- | --- |
| `basic` | 基础覆盖规划兜底能力。 | 否 |
| `shelf_aware` | 保守可用基线，使用 shelf-aware 原工程主链路和 UI 稳定保护。 | 否 |
| `shelf_aware_turn_cost` | 当前候选增强基线，融合 turn-cost 价值但不接入研究后处理大杂烩。 | 是 |
| `channel_topology_graph` | 通道拓扑图适配入口。 | 否 |

`shelf_aware_turn_cost` 不能替代 `shelf_aware` 成为唯一基线。工程上应长期保留至少两个可比较入口：一个保守基线，一个候选增强基线。

## 3. turn-cost 研究价值的分层

turn-cost 研究中有价值的内容不能统一写成“已经完整进入主流程”。当前应分成三类维护，避免把目标模型、诊断守卫和代码事实混成一层真值。

### 3.1 当前已进入主流程的内容

| 内容 | 当前代码承载 | 价值 |
| --- | --- | --- |
| 规则覆盖图 / repaired grid | `turn_cost_repaired_grid` 节点生成 profile 和 `CoverageGraphView` 静态图边界。 | 比纯局部路径点更容易保持覆盖点规则性，并为后续遍历提供稳定身份。 |
| 转角代价 | 候选评分中的 `turn_cost` 组件和转角约束。 | 让候选选择阶段直接考虑急转、折返和方向突变，而不是只在最终图上统计。 |
| 方向与 CTG 评分 | `local_direction_cost`、`local_lateral_cost`、`ctg_*` 评分组件。 | 对走廊、货架行、长通道和局部拓扑切换提供软偏置。 |
| 节点生成 provenance | `generation_mode`、`generation_status`、offset、endpoint alignment 等静态字段。 | 让路径质量问题能回到节点生成来源和受限修复行为。 |

### 3.2 当前作为诊断或验收守卫的内容

| 内容 | 当前定位 | 不能误解为 |
| --- | --- | --- |
| 线间距稳定性 | 验收指标、只读诊断和后续缺口判断依据。 | 当前已经有完整 lane family 优化模型。 |
| 覆盖率、窄通道、长跳跃和几何风险守卫 | delivery gate、批量验证和只读几何诊断。 | 规划阶段已经以硬约束形式统一优化这些指标。 |
| provenance 摘要 | diagnostics 和 artifact 用于解释路径来源。 | 所有路径段都已有完整 cell/move/coverage contribution 归因。 |

### 3.3 仍属于缺口或后续模型的内容

| 内容 | 当前缺口 | 入口 |
| --- | --- | --- |
| lane family / topology source | 只能作为设计方向和局部诊断依据，尚不能支撑激进重连或删线。 | `docs/03_缺口/01_当前缺口总表.md` |
| coverage contribution | 尚不能可靠判断每段路径的独立覆盖贡献。 | `docs/03_缺口/01_当前缺口总表.md` |
| 几何 swept footprint 硬约束 | 当前是只读诊断，尚未进入 planner cost 或可行性约束。 | `docs/01_主题/03_机器人几何与覆盖定义/` |

## 4. 已拒绝作为正式主流程的内容

以下内容可以作为历史证据或未来研究起点，但不能作为 UI 正式链路的一部分：

- 论文 official flow 直接替换工程 planner；
- HiGHS/Gurobi official 复现结果直接进入 UI；
- 全路径点吸附到 turn-cost 节点；
- 无约束 lane 删除；
- 批量 shortcut、batch reconnect 或局部窗口大规模重连；
- 单 case 参数特调；
- 把 geometry diagnostic 作为规划硬约束直接塞进主流程；
- 以 connector、fallback、postprocess 名义扩展新的补丁分支。

这些路线的共同问题是：局部指标可能改善，但难以证明全链路更自然、更泛化，也容易在覆盖率、长跳跃、线距或可读性上引入新问题。

## 5. 输入边界

本方案只消费当前工程已有预处理结果。它不独立重做地图预处理，不额外定义通用语义映射，不把 pass-only 区域、软代价区或地图清洗逻辑塞进算法本体。

如果预处理结果本身缺少必要信息，应该在预处理链路或输入契约中解决，而不是由 `shelf_aware_turn_cost` 私自补语义。

## 6. 输出边界

正式输出仍以 `CoveragePlanningResult` 为唯一对外真值。调试图、JSON、CSV、manifest 和 diagnostics 都是证据产物，不允许成为第二套路径真值。

输出质量至少需要关注：

- 覆盖率和窄通道覆盖；
- lane 过密、过疏、左右不平衡；
- 总转角、急转窗口、连续折线窗口；
- 长跳跃和不可行段；
- 几何只读诊断中的车体扫掠风险和真实清扫覆盖差异；
- 运行时间和参数可解释性。

## 7. 当前边界结论

`shelf_aware_turn_cost` 不是“把 shelfAware 结果再修一修”的后处理算法。它应当是一条从输入契约、覆盖图、遍历评分、路径实现到诊断产物都一致的候选增强主流程。

任何新改动如果只能解释为“再补一个例外”，应优先拒绝；如果能归入覆盖图建模、候选评估、路径实现边界或诊断契约，才允许进入设计讨论。
