# Coverage Planning 模块边界说明

## 1. 模块定位

`algorithms/coverage_planning` 是仓库中的正式一级算法工程，负责：

- 对外提供统一的覆盖规划 formal contract
- 根据场景适用性在不同 planner 之间做正式路由
- 承载可长期维护的区域覆盖 planner
- 通过正式 adapter 对接 `channel_topology_graph`

它不是：

- GUI 层
- 测试集工程
- 报告跑批目录
- `channel_topology_graph` 的实现承载层

## 2. 目录边界

### 2.1 `contracts/`

职责：

- 定义请求对象
- 定义结果对象
- 定义适用性判断相关对象
- 定义诊断信息对象

约束：

- 这里放正式 contract，不放具体 planner 算法实现
- 这里的字段口径要对外稳定，不能夹带 GUI 临时状态

### 2.2 `planners/`

职责：

- 承载具体 planner 实现
- 每个 planner 子目录负责自己的内部实现与依赖组织

当前正式 planner：

- `region_basic/`
  - 面向 room-like 的基础区域覆盖 planner
- `shelf_aware_guarded/`
  - 面向 aisle-like 默认 guarded 路线的 planner

约束：

- `planners/` 只承载算法实现，不负责场景路由决策
- 不要把 `channel_topology_graph` 实现整体迁入 `planners/`

### 2.3 `routing/`

职责：

- 计算适用性指标
- 进行场景分类
- 选择正式 planner
- 统一把 planner 输出收口成 formal result

约束：

- `routing/` 只负责“选哪个 planner”和“怎么收口结果”
- 不直接承载复杂 planner 主实现
- 不把 room segmentation 当作强依赖前置

### 2.4 `routing/adapters/`

职责：

- 对接不在本工程内部实现的正式算法工程
- 当前主要是 `channel_topology_graph_adapter.py`

约束：

- adapter 是边界层，不是实现迁移层
- adapter 负责参数翻译、结果收口和调用边界，不负责复制 CTG 内部逻辑

### 2.5 `planner_factory.py`

职责：

- 给调用方提供稳定创建入口
- 向 `maptools` 或顶层工具暴露受控的 planner 创建方式

约束：

- factory 只做创建与路由入口绑定
- 不把复杂算法决策、测试逻辑、报告写盘逻辑塞进 factory

## 3. 与其他子工程的边界

### 3.1 与 `maptools/` 的边界

`maptools/` 负责：

- GUI 交互
- 文件导入导出
- 用户配置与产品工作流

`algorithms/coverage_planning` 负责：

- planner contract
- routing
- planner 实现

边界约束：

- `maptools` 应通过稳定 contract / factory 调用 planner
- 不应直接 import planner 内部细节模块

### 3.2 与 `coverage_dataset/` 的边界

`coverage_dataset/` 负责：

- case 组织
- 模板展开
- 批量运行
- 报告治理

`algorithms/coverage_planning` 不负责：

- 管理 case 目录
- 管理 reports
- 保存 benchmark 资产

### 3.3 与 `channel_topology_graph/` 的边界

`channel_topology_graph/` 是独立正式算法工程，负责：

- aisle / shelf-like 拓扑覆盖主实现
- 自己的对象契约、阶段设计、support、smoke、baselines

`coverage_planning` 负责：

- 决定何时调用 CTG
- 通过 adapter 做正式边界接入

边界约束：

- 不能把 CTG 当作 `coverage_planning/planners/` 下的普通子实现
- 不能在 `coverage_planning` 中复制 CTG 规则形成第二套实现

## 4. 长期演化约束

- 新增 planner 时，先判断它是不是正式一级算法工程，还是 `coverage_planning` 内部的一个 planner
- 如果算法有独立对象契约、独立 docs、独立 smoke/baseline/support，优先评估是否应成为像 CTG 一样的独立算法工程
- 如果只是 room-like / mixed / aisle-like 路由下的一个可替换 planner，则应落在 `planners/`
- 不允许把历史原型或临时实验代码直接塞回 `python_ws` 风格目录
